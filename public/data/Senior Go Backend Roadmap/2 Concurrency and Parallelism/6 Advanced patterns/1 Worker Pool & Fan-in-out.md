# 1. Worker Pool & Fan-in/Fan-out

## Worker Pool (Пул воркеров)
Паттерн для ограничения количества одновременно выполняемых задач. 
**Зачем**: Предотвращение исчерпания ресурсов (память, файловые дескрипторы, лимиты внешних API).

### Статический воркер-пул
Количество горутин фиксировано. Слушают один общий канал задач.
```go
for w := 1; w <= numWorkers; w++ {
    go worker(jobs, results)
}
```

### Динамический воркер-пул (Staff уровень)
Пул, который увеличивает количество горутин при росте очереди и завершает лишние горутины при простое. Реализуется через комбинацию каналов и `atomic/mutex` счетчиков.

## Fan-out / Fan-in
- **Fan-out**: Распараллеливание одной задачи на множество горутин.
- **Fan-in**: Сбор результатов из множества горутин в один результирующий поток.

## Вопросы для интервью (Senior/Staff)

### Как правильно реализовать Fan-in для неопределенного количества отправителей?
Использовать `sync.WaitGroup` для отслеживания всех отправителей и запустить отдельную горутину-мерджер, которая будет читать из всех каналов и писать в один. Когда WaitGroup обнулится — мерджер закрывает выходной канал.

### В чем опасность классического паттерна Worker Pool с одним каналом задач?
Если обработка в воркерах идет с разной скоростью, быстрые воркеры могут простаивать, пока медленные блокируют очередь (если она была буферизована специфично). Также один общий канал становится точкой конкуренции (lock contention) за `hchan.lock`. На экстремальных нагрузках это может стать узким местом.

### Как реализовать приоритетную очередь задач в Worker Pool?
В Go `select` работает случайно, поэтому нельзя просто сделать `select { case high: ... case low: ... }`. Приоритезация достигается либо использованием нескольких пулов воркеров с разным весом, либо использованием внешней структуры данных (например, кучи/heap) под мьютексом для хранения задач вместо каналов.

### Что такое "Leaky Workers" и как их избежать?
Это воркеры, которые не завершились после остановки основной программы или завершения контекста. Решение: всегда дожидаться завершения через `sync.WaitGroup` и сигнализировать об остановке через `context.Context` или закрытие канала задач. Пул без механизма остановки — мина замедленного действия.
