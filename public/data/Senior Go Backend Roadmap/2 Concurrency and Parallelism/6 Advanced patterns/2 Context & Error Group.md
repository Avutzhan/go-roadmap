# 2. Context & Error Group

## context.Context
Стандартный механизм Go для управления временем жизни запроса, передачи сигналов отмены и метаданных по дереву вызовов.

### Методы и типы
- `WithCancel`: Явная отмена.
- `WithTimeout` / `WithDeadline`: Отмена по времени.
- `WithValue`: Передача метаданных (логи, ID запроса, токены). **Staff правило**: не использовать для передачи бизнес-параметров.

## sync/errgroup (x/sync)
Пакет, расширяющий возможности `sync.WaitGroup`. Он позволяет:
1. Остановить всю группу горутин, если хотя бы одна вернула ошибку.
2. Вернуть первую произошедшую ошибку.
3. Ограничивать количество активных горутин.

## Вопросы для интервью (Senior/Staff)

### Почему context.Value использует интерфейс{} в качестве ключа и как это сделать правильно?
Для предотвращения коллизий между разными пакетами. Если два пакета используют строку `"userID"`, они перезапишут данные друг друга. Решение: создавать свой приватный тип для ключа внутри пакета: `type key uint32; const UserKey key = 1`.

### Можно ли использовать контекст как замену каналам данных?
Нет. Контекст не предназначен для передачи полезной нагрузки (data flow). Его задача — сигнализация и передача сквозных метаданных (cross-cutting concerns). Использование контекста для данных делает код неявным и трудным для тестирования.

### Как errgroup помогает в реализации Graceful Shutdown?
Она идеально подходит для запуска нескольких серверов (например, HTTP и gRPC) в одном процессе. Если один сервер упал или контекст отменился, `errgroup` через связку с `context.WithCancel` отправит сигнал всем остальным компонентам на корректное завершение.

### В чем опасность context.Background() в глубоких слоях бизнес-логики?
Это разрывает цепочку отмены. Если верхний уровень (HTTP запрос) уже отменен, нижние слои, создавшие новый `Background()`, продолжат тратить ресурсы CPU и БД на работу, результат которой уже никому не нужен. Контекст должен всегда "прокидываться" сверху вниз.
