# 1. Happens-before & Memory Barriers

## Модель памяти Go (Memory Model)
Модель памяти определяет порядок, в котором операции записи в одной горутине становятся видимыми для операций чтения в другой. Без явной синхронизации компилятор и процессор могут переупорядочивать инструкции для оптимизации.

## Отношение "Happens-before"
Если событие **A** происходит "до" **B**, то результат **A** гарантированно виден в момент выполнения **B**.

### Гарантии в Go
1. **Init**: Инициализация пакета в одной горутине завершается "до" начала любой функции в этом пакете.
2. **Goroutine start**: Вызов `go f()` происходит "до" начала выполнения функции `f`.
3. **Channel send/recv**: 
   - Завершение записи в канал происходит "до" завершения соответствующего приема.
   - Закрытие канала происходит "до" получения zero value из него.
   - Прием из небуферизованного канала завершается "до" того, как отправитель закончит отправку (синхронизация).
4. **Locking**: `Unlock` мьютекса происходит "до" любого последующего `Lock` того же мьютекса.

## Вопросы для интервью (Senior/Staff)

### Зачем нужны Memory Barriers (барьеры памяти)?
Это инструкции процессора, которые запрещают переупорядочивание операций чтения/записи через границу барьера. В Go программист не расставляет их вручную, рантайм вставляет их неявно при использовании мьютексов, атомиков и каналов. Это гарантирует консистентность данных между кешами разных ядер CPU.

### Что такое Instruction Reordering?
Компилятор может поменять строки кода местами, если они не зависят друг от друга по данным. Пример:
```go
a = 1
b = 2
```
На уровне процессора `b = 2` может выполниться раньше `a = 1`. Если другая горутина ждет `b == 2`, чтобы прочитать `a`, она может увидеть "мусор" или `0`. Модель памяти Go запрещает такие опасные оптимизации только при использовании синхронизации.

### Почему чтение без мьютекса — это всегда риск, даже если переменная "простая" (например, число)?
Из-за иерархии кэшей. Изменение переменной в одном ядре CPU может долго находиться в его L1/L2 кэше и не быть "сброшенным" (flushed) в общую память L3/RAM. Мьютексы и каналы форсируют синхронизацию кэшей между ядрами.

### Гарантирует ли модель памяти Go строгую консистентность (Strict Consistency)?
Нет, Go гарантирует только **Sequential Consistency** при соблюдении правил синхронизации. Без них поведение программы считается неопределенным (Undefined Behavior), и вы можете столкнуться с "фантомными чтениями", когда значение то появляется, то исчезает из видимости.
