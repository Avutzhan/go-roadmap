# 2. Graceful Shutdown

## Определение
Graceful Shutdown — это процесс корректного завершения работы приложения, при котором:
1. Останавливается прием новых клиентских соединений.
2. Текущие задачи дорабатываются до конца (в пределах таймаута).
3. Сохраняется состояние (если нужно).
4. Закрываются соединения с БД, брокерами сообщений и логами.

### Механика в Go
Основана на прослушивании системных сигналов ОС (`SIGINT`, `SIGTERM`) через канал `os/signal`.

## Вопросы для интервью (Senior/Staff)

### Как реализовать Graceful Shutdown для HTTP сервера?
Использовать метод `server.Shutdown(ctx)`. Он закрывает слушающий сокет и ждет, пока все активные соединения закроются или пока не истечет контекст. Важно запустить этот метод в отдельной горутине или в основном потоке после получения сигнала.

### Почему waitGroup.Wait() может быть опасен при выходе приложения?
Если одна из горутин зависла (утечка), вызов `Wait()` заблокирует завершение процесса навсегда. В Kubernetes это приведет к тому, что контейнер будет убит принудительно по `SIGKILL` через 30 секунд. Решение: использовать `Wait()` внутри `select` с таймаутом.

### Что делать с сообщениями из очереди (RabbitMQ/Kafka) при выключении?
Нужно перестать делать `Consume()`, дождаться подтверждения (`Ack`) для текущих сообщений и закрыть соединение. Если сообщение не успело обработаться — его следует оставить в очереди (не подтверждать), тогда другой инстанс подхватит его.

### Зачем нужен второй сигнал (Force Kill)?
Если приложение застряло в процессе Graceful Shutdown, пользователь должен иметь возможность нажать `Ctrl+C` второй раз для немедленного выхода. В коде это реализуется через отслеживание количества входящих сигналов или через таймер на контексте остановки.
