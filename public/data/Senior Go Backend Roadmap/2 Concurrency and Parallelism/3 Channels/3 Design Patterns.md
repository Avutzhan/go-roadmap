# 3. Design Patterns with Channels

## Unbuffered vs Buffered: Философия
- **Unbuffered (Размер 0)**: Гарантия синхронизации. Отправитель знает, что получатель реально принял данные. Это "сильное" рукопожатие.
- **Buffered (Размер > 0)**: Разрывает жесткую связь по времени. Позволяет сглаживать пики нагрузки. Отправитель не знает, когда данные будут обработаны.

## Паттерны для Staff уровня

### 1. Or-channel (Сигнал от любого из многих)
Создание одного канала, который закрывается (выстреливает), когда закрывается любой из переданного набора каналов. Полезно для ожидания завершения первой из множества асинхронных задач.

### 2. Pipeline (Конвейер)
Цепочка этапов обработки, соединенных каналами. Позволяет эффективно использовать все ядра CPU, так как каждый этап работает параллельно над своим куском данных.

### 3. Bridge (Канал каналов)
Разворачивание стрима каналов (например, `<-chan <-chan int`) в один плоский канал.

## Вопросы для интервью (Senior/Staff)

### В чем опасность больших буферов в каналах?
Большой буфер может маскировать проблемы с производительностью (утечки памяти или медленных потребителей). Также он усложняет Graceful Shutdown: при падении процесса данные в буфере будут потеряны, если только они не сохранены во внешнем хранилище. Правило: размер буфера должен быть либо 1, либо обоснован конкретным профилем нагрузки.

### Как реализовать таймаут для операции в канале?
Через `select` и `time.After`:
```go
select {
case res := <-ch:
    // успех
case <-time.After(time.Second):
    // таймаут
}
```
**Нюанс для Senior**: В высоконагруженных циклах лучше использовать `time.Timer` и переиспользовать его через `Reset()`, чтобы не создавать тысячи короткоживущих каналов и не нагружать GC.

### Что такое Nil-channel trick в select?
Если в `select` установить канал в `nil`, этот кейс больше никогда не сработает (чтение/запись в nil блокируют навсегда, а select просто игнорирует заблокированные кейсы). Это используется для динамического отключения ветки `select` без выхода из цикла.

### Как передать владение каналом?
Лучшая практика — возвращать канал только для чтения (`<-chan int`) из функции-провайдера. Это четко определяет ответственность: только провайдер может писать в канал и закрывать его. Потребитель только читает. Это предотвращает паники от записи в закрытый канал.
