# 1. GMP Architecture: Deep Dive

## Модель GMP
Рантайм Go управляет горутинами через три ключевых сущности, которые обеспечивают эффективное использование ядер CPU без прямого создания тысяч потоков ОС.

### G (Goroutine)
Это легковесный поток. Логически это структура в памяти (`runtime.g`), которая содержит:
- **Стек**: Начальный размер 2KB. Стек горутины растет и уменьшается динамически.
- **Состояние**: `_Gidle`, `_Grunnable`, `_Grunning`, `_Gsyscall`, `_Gwaiting`.
- **Контекст**: Указатель на текущую инструкцию (PC) и переменные в регистрах для переключения контекста.

### M (Machine / OS Thread)
Это физический поток операционной системы (`runtime.m`).
- Выполняет код горутины.
- Требует наличия **P** для исполнения кода Go.
- Максимальное количество потоков по умолчанию — 10,000 (но обычно их гораздо меньше, около 50-100).

### P (Processor)
Это логический ресурс (`runtime.p`), контекст планировщика.
- Определяет количество параллельно работающих потоков.
- Количество P задается через `GOMAXPROCS`.
- Содержит **локальную очередь (LRQ)** на 256 горутин.

## Взаимодействие
Для выполнения горутины рантайм должен сопоставить **G** с **M**, а **M** должен владеть **P**.
Уравнение работы: `M + P -> G`.

## Вопросы для интервью (Senior/Staff)

### В чем разница между локальной (LRQ) и глобальной (GRQ) очередями?
- **LRQ (Local Run Queue)**: Находится внутри каждого **P**. Доступ к ней не требует блокировок мьютексом для текущего M, что обеспечивает высочайшую производительность.
- **GRQ (Global Run Queue)**: Общая очередь для всех процессоров. Доступ к ней защищен мьютексом. Сюда попадают горутины, которые были недавно созданы, если LRQ воркера переполнена, или те, что "украдены" планировщиком при перебалансировке. Планировщик заглядывает в GRQ каждые 61 тик, чтобы избежать голодания глобальных задач.

### Что происходит, когда горутина создает новую горутину (go func)?
Она пытается поместить ее в **Next-G** — специальный слот в **P**, который обеспечивает выполнение новой горутины сразу после текущей (LIFO-оптимизация для локальности данных). Если слот занят, она отправляется в LRQ текущего процессора. Если и LRQ полна (256 элементов), половина очереди перемещается в GRQ.

### Какова роль поля schedtick в структуре P?
Это счетчик тиков планировщика. Он используется для определения момента, когда нужно проверить глобальную очередь (каждый 61-й вызов функции `findrunnable`). Это критично для предотвращения ситуации, когда горутины в глобальной очереди никогда не получат времени CPU из-за "шумных" локальных очередей.
