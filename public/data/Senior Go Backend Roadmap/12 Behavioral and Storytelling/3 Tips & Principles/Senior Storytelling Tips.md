# 🧠 2. Senior Storytelling: Interview Performance Tips

## 📌 Что это такое

Технические знания нужны чтобы пройти screening. Но Senior/Staff позицию получает тот кто умеет **убедительно рассказывать** о своём опыте. Интервьюер часто принимает решение в первые 5 минут istory.

---

## 🔬 Принципы Senior Storytelling

### Принцип 1: Данные > слова

```
❌ Weak: "Мы сильно улучшили производительность системы"
✅ Strong: "p99 Latency упала с 820мс до 95мс, что эквивалентно 
            снижению серверных расходов на $8K/месяц"

❌ Weak: "Команда стала работать быстрее после моего решения"
✅ Strong: "Deployment frequency выросла с 2x/month до ежедневно.
            Change failure rate: с 35% до 8% за квартал."

Правило: каждое достижение = число + контекст + бизнес-impact
Если не помнишь точные числа: "около X" лучше чем вообще без числа
Никогда не выдумывай цифры — опытный интервьюер проверит логику
```

### Принцип 2: "Я" не "Мы" — но с уважением к команде

```
❌ "Мы переписали всю систему"  → непонятна твоя роль
❌ "Я в одиночку спас компанию" → звучит неправдиво
✅ "Команда работала над X. Моей ролью было:
    (1) Дизайн архитектуры cache layer
    (2) Code review и mentoring двух junior инженеров
    (3) Координация с DB командой для schema migration.
    Мой личный вклад: снижение query time с 2s до 120ms через правильные индексы."
```

### Принцип 3: Структура Story Arc

```
Хорошая история имеет arc (дугу):

BEFORE: [описание проблемы с данными]
  "Наш payment service обрабатывал 3000 RPS, при пиках — деградация до 12s latency"

TENSION: [почему это было сложно/интересно]
  "Нельзя было просто добавить серверов — bottleneck был в архитектуре:
   синхронные calls по цепочке 4 downstream сервисов"

ACTION: [твоя конкретная роль — самая большая часть]
  "Я провёл distributed tracing анализ (Jaeger), нашёл bottleneck в Payment Gateway.
   Предложил async pattern через Kafka.
   Написал RFC, провёл 3 alignment сессии с командами.
   Лично имплементировал circuit breaker и outbox pattern"

RESULT: [измеримый outcome]
  "p99: 12s → 85ms. Zero downtime migration за 6 недель.
   Следующий квартал: 0 latency-related incidents."

LESSON: [что ты из этого извлёк — только для best stories]
  "Научился что distributed tracing должен быть с дня 1, не после проблемы"
```

### Принцип 4: Калибровка под уровень интервьюера

```
Разговариваешь с Engineering Manager:
→ Фокус на: бизнес impact, работу с командой, процессы

Разговариваешь с Staff Engineer:
→ Фокус на: технические trade-offs, архитектурные решения, scalability

Разговариваешь с Principal/Distinguished:
→ Фокус на: org-wide impact, influence, system thinking, long-term vision

Как понять кто перед тобой? Первые 2 минуты — слушай КАК они задают вопросы.
Technical depth в вопросе = technical depth в ответе
```

### Принцип 5: Время ответа

```
Ideal answer timing:
- STAR ответ: 3-5 минут
- Situation + Task: 20-30 секунд (контекст, не история)
- Action: 70% времени — это твоя ценность
- Result: 30-60 секунд — конкретные числа

Если прошло > 5 минут: "Это основное, есть вопросы или пойдём глубже?"

Признаки что ты теряешь интервьюера:
- Он начинает что-то писать (уходит в свои мысли)
- "Мм-гм" без follow-up вопросов
- Смотрит в сторону

Спаси: "Хочу убедиться что даю нужный level detail. Interested в архитектуре или больше в процессе?"
```

## 🔥 Опасные паттерны которых надо избегать

```
❌ Hero story: "Я один спас весь продукт пока команда спала"
   → Поднимает вопрос: "Почему команда не могла без тебя?"

❌ Blame game: "PM дал плохие требования, тестировщики пропустили баги"
   → Интервьюер думает: "Этот человек не возьмёт ответственность"

❌ Overly technical details для non-technical interviewer:
   "Использовал GMP scheduler с work stealing и preemption points..."
   → Потерял audience

❌ "Мы сделали X" без твоей конкретной роли:
   → "А что именно ты делал?"

❌ Нет числа в результате:
   "Стало намного быстрее" → "Насколько быстрее?"
```

## 💬 Шаблоны открывающих фраз

```
Для technical story:
"Это было в [2023], когда я работал Old Company над [domain].
 Мы столкнулись с [проблемой] — конкретно [измеримый симптом].
 Моей ответственностью было [конкретная роль]."

Для leadership/influence story:
"Расскажу о ситуации где мне нужно было[убедить команду/принять сложное решение].
 Контекст: [30 секунд].
 Моё решение было [Action] потому что [логика].
 В результате [outcome]."

Для failure/learning story:
"Поделюсь ситуацией где я допустил ошибку и что из этого вынес.
 [Ситуация + что пошло не так].
 Мой прямой вклад в проблему был [честно].
 Что я изменил после: [конкретные действия]."
```

## ❓ Вопросы для интервью (Senior/Staff)

### Как ответить на "Расскажи о себе" на Senior позиции (< 2 минут)?

Структура: текущая роль (1 предложение) + 2-3 ключевых достижения с числами + почему интересна эта позиция. Не biography. "Сейчас я Lead Backend Engineer в X, отвечаю за payment platform обрабатывающую 5M транзакций/день. За последний год: уменьшил p99 latency платежей в 3x, мигрировал payment-service из монолита без downtime, вырастил двух junior до mid инженеров. Интересует эта роль потому что [конкретно о компании]."

### Как ответить на "Какая твоя слабость"?

Настоящая слабость (не "я слишком перфекционист"). Покажи awareness и рост: "Раньше я слишком долго работал над дизайном перед началом реализации — analysis paralysis. Научился использовать time-boxed spikes: 4 часа исследования → begin. За 2024 год deployment frequency нашей команды выросла в 4x, частично из-за этого изменения."
