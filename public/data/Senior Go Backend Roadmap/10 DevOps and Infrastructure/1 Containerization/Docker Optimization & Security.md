# 1. Docker Optimization & Security for Go

## Оптимизация Docker образов для Go
Go компилируется в статически связанный бинарный файл, что позволяет делать образы экстремально маленькими.

### Multi-stage Builds (Многоэтапная сборка)
Это стандарт де-факто. Мы используем один образ для сборки (с полным Go SDK) и другой — минимальный — для запуска.
```dockerfile
# Stage 1: Build
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY . .
RUN go build -o main .

# Stage 2: Run
FROM alpine:latest  
COPY --from=builder /app/main /main
CMD ["/main"]
```

### Distroless образы
Еще более безопасный вариант, чем Alpine. Не содержит командной оболочки (shell), пакетного менеджера и других утилит. Это уменьшает "поверхность атаки" (Attack Surface).

## Вопросы для интервью (Senior/Staff)

### В чем преимущество статической линковки Go для Docker?
Вам не нужно тащить в образ зависимости типа `libc` или интерпретаторы. Весь ваш код и его библиотеки упакованы в один файл. Это позволяет создавать образы размером 10-20МБ вместо 500МБ (как в Java или Python).

### Как ускорить сборку Docker образа (Caching Layers)?
Docker кэширует каждый слой. Чтобы не перекачивать все зависимости (`go mod download`) при каждом изменении кода, нужно сначала копировать только `go.mod` и `go.sum`, запускать `go mod download`, и только потом копировать остальной исходный код.

### Зачем использовать не-root пользователя внутри контейнера?
По умолчанию процессы в контейнере работают от root. Если злоумышленник взломает ваше приложение и найдет уязвимость в рантайме контейнера, он может получить доступ к хост-системе с правами root. Использование `USER nonroot` — база безопасности.

### Как сканировать образы на уязвимости?
Использовать инструменты типа **Trivy** или **Snyk**. Они проверяют слои образа на наличие известных CVE (уязвимостей) в системных библиотеках (даже в Alpine они есть). Это должно быть частью CI/CD пайплайна.
