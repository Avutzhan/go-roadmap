# 1. Linux CLI & Git Workflows

## Linux для Бэкенд-инженера
Если сервис упал, вы должны уметь войти на сервер и понять, что происходит.
- **Анализ нагрузки**: `htop` (CPU), `free -h` (Memory), `iostat` (Disk I/O).
- **Сеть**: `tcpdump` (перехват пакетов), `lsof -i :8080` (кто занял порт), `curl -v` (детали HTTP обмена).
- **Логи**: `journalctl -u my-service -f`.

## Git Workflows
- **Trunk Based Development**: Все коммитят в `main` (через короткоживущие ветки). Требует сильных тестов и Feature Flags. Стандарт для High-performing команд.
- **Git Flow**: Сложная иерархия (develop, master, release, hotfix). Медленнее, но строже.

## Вопросы для интервью (Senior/Staff)

### В чем разница между `git merge` и `git rebase`?
- **Merge**: Создает "merge commit" и сохраняет историю в хронологическом порядке. Хорошо для понимания, когда фича влилась.
- **Rebase**: Переписывает историю, как будто вы писали фичу поверх самого свежего кода `main`. Делает историю линейной и чистой.
**Senior выбор**: Rebase для своих локальных веток перед пушем, Merge для вливания в основную ветку (если это политика компании).

### Как найти процесс, который "съел" всю оперативную память на сервере?
Команда `top` или `htop`, нажать `M` для сортировки по памяти. Также полезно `ps aux --sort=-%mem`.

### Что делает команда `strace`?
Она перехватывает системные вызовы (syscalls), которые совершает программа (открытие файлов, запись в сокеты). Это "тяжелая артиллерия" дебага, когда само приложение ничего не пишет в логи, но почему-то не работает.

### Почему `git reset --hard` опасен?
Он безвозвратно удаляет все незафиксированные изменения в рабочей директории и индексе. Если вы случайно сделали `hard reset` не на ту ветку — данные будет крайне сложно (иногда невозможно) восстановить. Всегда проверяйте текущую ветку перед этой командой.
