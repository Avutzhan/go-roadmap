# 1. LRU Cache & Trie

## LRU Cache (Least Recently Used)
Структура данных, которая вытесняет самые старые (давно не использованные) элементы при достижении лимита.
- **Реализация**: Хеш-мапа + Двусвязный список.
- **HashMap**: Дает доступ к узлам списка по ключу за **O(1)**.
- **Doubly Linked List**: Позволяет быстро перемещать элемент в начало (при использовании) и удалять из конца (при вытеснении) за **O(1)**.

## Trie (Prefix Tree)
- Позволяет эффективно искать слова по префиксу.
- Используется в высокопроизводительных Router-ах (например, в некоторых Go HTTP-фреймворках для сопоставления путей типа `/api/v1/user/:id`).

## Вопросы для интервью (Senior/Staff)

### Почему в LRU Cache используется именно двусвязный список, а не односвязный?
Потому что в двусвязном списке при удалении узла мы имеем доступ к его предыдущему соседу за **O(1)**. В односвязном нам пришлось бы искать соседа с начала списка за O(N). Это критично для поддержания константной сложности операций кеша.

### Как сделать LRU Cache потокобезопасным (Thread-safe) в Go?
Обернуть операции в `sync.Mutex`. Но на High-load одномьютексный кеш станет бутылочным горлышком (Lock Contention). 
**Staff решение**: Шардирование кеша (разделение на несколько мап со своими мьютексами по хешу ключа).

### В чем преимущество Trie перед HashMap при поиске префиксов?
HashMap умеет искать только полные совпадения. Если вам нужно "найти всё на букву A", в мапе придется просканировать все ключи **O(N)**. В Trie вы просто спускаетесь до узла 'A' и забираете все его поддеревья.

### Какую память потребляет Trie и как её оптимизировать?
Trie может потреблять много памяти из-за указателей в каждом узле.
**Оптимизация**: **Radix Tree (Compact Trie)** — узлы с одним ребенком схлопываются в один (например, вместо "a"->"p"->"p"->"l"->"e" будет один узел "apple"). Это стандарт для HTTP роутеров (например, Gin использует Radix Tree).
