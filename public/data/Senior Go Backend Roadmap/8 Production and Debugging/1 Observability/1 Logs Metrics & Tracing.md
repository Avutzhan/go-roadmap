# 1. Logs, Metrics & Tracing: The Observability Pillar

## Логирование (Logs)
- **Structured Logging**: В продакшене логи должны быть в формате JSON (разбираются ELK/Grafana Loki).
- **Performance**: Популярные библиотеки Go:
    - `uber-go/zap`: Чрезвычайно быстрый, поддерживает структурирование.
    - `rs/zerolog`: Фокус на нулевых аллокациях (zero-alloc), очень легкий.
- **Staff уровень**: Никогда не логируйте чувствительные данные (пароли, PII). Используйте уровни логирования (Debug, Info, Warn, Error) с умом, чтобы не "заспамить" диски.

## Метрики (Metrics)
Обычно реализуются через **Prometheus**.
- **Counter**: Только растет (количество запросов, ошибок).
- **Gauge**: Может расти и падать (текущее количество горутин, использование памяти).
- **Histogram**: Распределение значений (Latency). Позволяет считать перцентили (p95, p99).
- **Summary**: Похож на гистограмму, но квантили считаются на стороне приложения (тяжелее для CPU).

## Трейсинг (Tracing)
- **Зачем**: Позволяет отследить путь одного запроса через 20 микросервисов.
- **OpenTelemetry (OTel)**: Современный стандарт. 
- **Spans & Context**: Каждый вызов — это Span. Контекст передается через HTTP заголовки (`traceparent`).

## Вопросы для интервью (Senior/Staff)

### В чем разница между логами и метриками в плане стоимости и пользы?
Метрики очень дешевы: это просто числа в памяти. Их можно хранить годами. Логи очень дороги: хранение текста требует много места и индексации. 
**Правило**: Метрики — чтобы узнать, **что** случилось (алерт). Логи — чтобы понять, **почему** это случилось (дебаг).

### Почему для измерения Latency нужно использовать перцентили (p99), а не среднее значение (Average)?
Среднее значение скрывает "хвосты". Если у 99 пользователей запрос прошел за 10мс, а у одного — за 10 секунд, среднее будет ~110мс. Это выглядит неплохо, но один пользователь страдает. p99 покажет реальную картину для худших случаев.

### Как реализовать контекстное логирование в Go?
Через `context.Context`. Мы сохраняем `request_id` или `trace_id` в контекст в middleware и затем извлекаем его при каждом вызове логгера. Это позволяет сопоставить все логи одного запроса.

### Что такое "Distributed Context Propagation"?
Это передача метаданных (traceID, baggage) между сервисами. В Go это делается через `otel.GetTextMapPropagator().Inject(ctx, ...)` при отправке запроса и `Extract` при получении. Без этого цепочка трейсинга разорвется.
