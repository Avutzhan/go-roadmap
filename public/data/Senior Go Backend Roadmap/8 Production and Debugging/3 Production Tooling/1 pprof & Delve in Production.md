# 1. pprof & Delve in Production

## pprof под нагрузкой
Go позволяет снимать профили прямо с работающего сервиса через HTTP.
- `/debug/pprof/profile`: CPU профиль (обычно за 30 секунд).
- `/debug/pprof/heap`: Профиль кучи (кто занимает память).
- `/debug/pprof/allocs`: История аллокаций.
- `/debug/pprof/block` / `mutex`: Поиск блокировок.

**Staff совет**: Снимать профиль в продакшене безопасно, оверхед обычно составляет 1-5%. В высоконагруженных системах можно настроить автоматическое снятие профиля при превышении порога CPU/Memory (Continuous Profiling - Pyroscope, Datadog).

## Delve (dlv)
Стандартный отладчик для Go.
- **Remote Debugging**: Позволяет подключиться к удаленному (или контейнеризированному) процессу и расставить точки останова.
- **core dumps**: Если приложение упало с паникой, dlv может открыть файл "слепка памяти" (core) и показать состояние всех переменных в момент краха.

## Вопросы для интервью (Senior/Staff)

### Как найти "горячий путь" (hot path) в коде с помощью pprof?
Снять CPU профиль во время нагрузки, открыть в Google Chrome (`go tool pprof -http=:8080 cpu.prof`) и посмотреть на **Flame Graph**. Самые широкие полоски — это функции, которые потребляют больше всего времени процессора. Нужно копать в них.

### В чем опасность включения `/debug/pprof/` в публичном доступе?
Это огромная дыра в безопасности. Злоумышленник может:
1. Увидеть структуру вашего кода.
2. Прочитать чувствительные данные из памяти (через кучу).
3. Положить ваш сервис, запустив тяжелые профилировщики одновременно (DoS).
**Решение**: pprof должен быть доступен только по внутреннему IP или защищен паролем.

### Как отладить Deadlock, если приложение "зависло" и не отвечает на HTTP?
Отправить процессу сигнал `SIGQUIT` (`kill -3 PID`). Go выдаст в консоль (`stderr`) полный стек-трейс всех горутин. Там будет написано, кто какую блокировку ждет. Это "последний шанс" дебага.

### Что такое "Stop The World" (STW) во время профилирования?
Некоторые виды профилирования (особенно детальный heap profile с обходом всех объектов) могут на короткое время приостанавливать выполнение программы для получения консистентного слепка данных. На огромных кучах (64ГБ+) это может вызвать заметный скачок Latency.
