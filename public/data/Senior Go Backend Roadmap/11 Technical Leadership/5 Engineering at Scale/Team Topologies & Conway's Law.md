# 🧠 2. Platform Engineering, Scale & Conway's Law

## 📌 Что это такое

Staff-инженер думает не только о своём сервисе, но о том, как **вся инженерная организация** производит и поддерживает программное обеспечение. Conway's Law и Team Topologies — два фундаментальных закона, которые определяют архитектуру систем.

---

## 🔬 Глубокий разбор (Senior/Staff)

### Conway's Law — архитектура отражает организацию

> *"Any organization that designs a system will produce a design whose structure is a copy of the organization's communication structure."* — Melvin Conway, 1967

```
Пример: Компания с 3 командами → создаёт компилятор из 3 этапов

Frontend Team    → UI-тяжёлый монолит
Backend Team     → API-first сервис
DB Team          → data-oriented микросервис

Если хочешь другую архитектуру — меняй организацию.
```

**Обратный Conway**: *Inverse Conway Maneuver* — сначала определи желаемую архитектуру, потом реорганизуй команды под неё.

### Team Topologies: 4 типа команд

```
1. Stream-aligned Team
   - Владеет end-to-end продуктовым потоком (Order Team, Payment Team)
   - Может деплоить независимо
   - Цель: быстрый flow value → users

2. Enabling Team
   - Помогает stream-aligned командам освоить новые практики
   - Временная: настроила → ушла
   - Пример: SRE Team, DevOps Enablement

3. Complicated Subsystem Team
   - Владеет сложной технической областью (ML Platform, Cryptography)
   - Маленькая, высококвалифицированная
   - Снижает cognitive load у stream-aligned команд

4. Platform Team
   - Создаёт Internal Developer Platform (IDP)
   - Другие команды используют как самообслуживание
   - Пример: Infrastructure Platform, Data Platform
```

### Cognitive Load: ключевая метрика Staff-инженера

```
Для Staff важно понять: команда переегружена?

Признаки перегруза:
- Новый сервис требует знания 15+ технологий
- Incident resolve занимает часы из-за неизвестных систем
- Onboarding нового члена > 3 месяцев

Staff-решение: уменьшить cognitive load через:
- Golden path (стандартные шаблоны)
- Abstractions: Kafka как "send message API" без знания internals
- Platform services: единый observability, CI/CD, secrets management
```

### Масштабирование инженерной организации

```
Закон Данбара (Dunbar's Number):
- ~5: тесная команда (full communication)
- ~15: расширенная команда (high trust)
- ~50: группа (частичное знакомство)
- ~150: организация (нужны процессы)

При > 150 инженеров:
→ Нужны tech forums / guilds (горизонтальное выравнивание)
→ Open source внутренние компоненты (Inner Source)
→ RFC процесс для кросс-командных решений
→ Technical Strategy документ
```

### Inner Source: OpenSource практики внутри компании

```
Модель: команда-владелец принимает PR от других команд
Правила:
1. Trusted Committers: члены команды с правом merge
2. Contributing Guidelines: как вносить изменения
3. Public backlog: другие команды видят roadmap и баги
4. "Consumer-driven contracts": изменения не ломают downstream

Пример: Platform Library для tracing/metrics — все команды вносят фикс или feature,
платформа reviews и merge, все пользуются.
```

## 🔥 Реальные боевые кейсы

- **Conway's Law в действии**: 2 отдела → 2 несовместимых API → microservices вынуждены синхронизироваться = нарушение decoupling
- **Inverse Conway**: руководитель хотел event-driven архитектуру → сначала создал Events Platform Team → команды сами стали делать event-driven
- **Cognitive Load metric**: новый инженер не мог on-call в течение 6 месяцев → диагноз: 23 репо с разными фреймворками → решение: golden path
- **Inner Source win**: команда баз данных создала connection pool lib, приняла 12 PR от 5 команд за квартал

## 💬 Как отвечать на интервью

> "Conway's Law говорит: ваша архитектура — зеркало вашей оргструктуры. Если хотите независимую микросервисную архитектуру — нужны независимые команды с end-to-end ownership. Inverse Conway Maneuver: сначала определяешь желаемую архитектуру, потом перестраиваешь команды. На Staff-уровне я думаю о cognitive load команд — если onboarding занимает месяцы, система слишком сложна. Platform Engineering уменьшает этот load через golden paths и self-service IDP."

## ❓ Вопросы для интервью (Senior/Staff)

### Как организовать RFC process для межкомандных архитектурных решений?

RFC (Request for Comments): 1) Автор пишет proposal в Google Doc/Confluence. 2) Период комментирования ~2 недели. 3) Tech Forum синхронно обсуждает. 4) Decision owner принимает решение с записью обоснования. 5) Хранится в ADR репо. Ключевое: RFC не заменяет Decision — в конце кто-то должен принять ответственность.

### Как Platform Team измеряет свой успех?

Не через технические метрики, а через **customer satisfaction**: NPS разработчиков, время от "идея" до "в production" (DORA метрики), adoption rate IDP. Если stream-aligned команды не пользуются платформой — она не решает их проблемы.

### Что такое DORA метрики и почему они важны для Staff?

DORA (DevOps Research and Assessment):
- **Deployment Frequency**: как часто деплоим (Elite: несколько раз в день)
- **Lead Time for Changes**: время от commit до production (Elite: < 1 часа)
- **Change Failure Rate**: % деплоев, вызвавших incident (Elite: < 5%)
- **Time to Restore Service**: время восстановления при incident (Elite: < 1 часа)

Staff-инженер использует эти метрики для обоснования инвестиций в автоматизацию.

# 1. Team Topologies & Conway’s Law

## Закон Конвея (Conway’s Law)
> "Организации, проектирующие системы, вынуждены создавать проекты, которые являются копиями структур коммуникаций этих организаций."
Если у вас 4 команды, вы скорее всего получите архитектуру из 4 основных сервисов.
**Staff совет**: Используйте "Reverse Conway Maneuver" — сначала спроектируйте идеальную архитектуру системы, а затем подстройте структуру команд под неё.

## Командные топологии (Team Topologies)
1. **Stream-aligned team**: Продуктовая команда, отвечающая за конкретный поток ценности (например, "Корзина заказов").
2. **Platform team**: Создает внутренние сервисы (инфраструктура, CI/CD, логирование), чтобы снизить нагрузку на продуктовые команды.
3. **Enabling team**: Эксперты (например, по безопасности или архитектуре), которые обучают другие команды.
4. **Complicated-subsystem team**: Глубокие спецы в узкой области (видео-кодеки, сложный поиск).

## Вопросы для интервью (Senior/Staff)

### Как структура команд влияет на количество инцидентов и скорость разработки?
Размытые границы ответственности ведут к конфликтам в коде и "бесхозным" сервисам. Четкое разделение по **Team Topologies** снижает когнитивную нагрузку и позволяет каждой команде двигаться независимо, не ожидая согласований от всех вокруг.

### Что такое "Лидерство без полномочий" (Leading without Authority)?
Умение направлять технологическое развитие компании или нескольких команд, не являясь их формальным руководителем. Это делается через:
1. Экспертность (ваше мнение ценят).
2. Помощь другим (вы полезны).
3. Вдохновение и аргументацию (за вашими идеями хотят идти).

### Как RFC (Request for Comments) помогает синхронизировать несколько команд?
Это открытый процесс обсуждения. Любой может прочитать и оставить комментарий к вашей идее. Это снимает "сюрпризы" при интеграции и позволяет выявить проблемы на этапе проектирования, объединяя опыт десятков инженеров.

### В чем опасность "Shared Library" (Общих библиотек) между командами?
Сильная связность (Tight Coupling). Изменение в общей библиотеке может сломать 20 сервисов разных команд. Senior-инженеры стараются делать библиотеки либо очень маленькими и стабильными, либо предоставлять функционал через API/Services.
