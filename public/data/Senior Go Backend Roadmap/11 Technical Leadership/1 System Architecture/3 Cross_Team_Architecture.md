# 11.7 Cross-Team Architecture (Staff Level)

На уровне Staff Engineer ваша зона видимости выходит за рамки одного микросервиса или даже одной команды. Вы начинаете проектировать контракты между доменами (Bounded Contexts) и департаментами.

## 1. Ownership Boundaries (Границы ответственности)
Главная проблема в больших компаниях — "кто за это отвечает?".
- Вы должны четко определять, где заканчивается ответственность вашей системы и начинается чужая.
- В микросервисной архитектуре это означает выстраивание Domain-Driven Design (DDD) контекстов. Ошибка здесь приводит к распределенному монолиту, где команды блокируют друг друга при каждом релизе.
- **Shared infrastructure**: Кто владеет общей БД? Ответ: никто не должен разделять БД (Shared Database Anti-pattern), сервисы общаются по API (Integration Contracts).

## 2. Dependency Management на уровне организации
- **Upstream / Downstream dependencies**: Если ваш сервис (Downstream) зависит от сервиса платформы (Upstream), как вы обеспечиваете отказоустойчивость?
- На уровне Стаффа вы применяете паттерны: **Circuit Breaker** (предотвратить лавинный сбой), **Fallback** (вернуть кэш или дефолтное значение), **Bulkheads** (разделить пулы коннектов, чтобы один упавший сервис не потопил остальные).
- Управление версионированием: Запрещено ломать обратную совместимость API (Backward Compatibility).

## 3. Integration Contracts & API Governance
- **API First Design**: Контракты (Swagger/OpenAPI, gRPC/Protobuf) пишутся до кода. Это позволяет командам работать параллельно (Frontend делает моки, Backend пишет логику).
- **Consumer-Driven Contracts (CDC)**: Обеспечение того, что провайдер API (сервер) случайно не сломает клиентов. Существуют фреймворки типа Pact, которые валидируют контракты во время CI/CD.
- Введение стандартов (Governance): Унификация названий полей, пагинации (Cursor vs Offset), обработки ошибок (RFC 7807 Problem Details). Решения фиксируются в инженерном справочнике.

## 💬 Вопросы для интервью

1. **Две команды разрабатывают фичи, которым нужны одни и те же данные о пользователе. Как вы их интегрируете без создания конфликтов?**
   *Ответ: Я не дам им лезть напрямую в чужую БД. Я организую Event Sourcing / Change Data Capture или предложу создать GraphQL/API Gateway слой, и определю строгие Integration Contracts.*

2. **Как вы обновляете общий API, который используется 5 другими командами, без даунтайма их сервисов?**
   *Ответ: Использую паттерн Expand & Contract. Добавляю новый эндпоинт/поле (версия v2), даю командам время мигрировать. Ставлю старый API в Deprecated, отслеживаю его использование по логам, и только когда трафик будет 0, удаляю v1.*
