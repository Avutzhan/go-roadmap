# 🧠 1. Code Review Culture & Architecture Decision Records

## 📌 Что это такое

Code Review (CR) на Senior/Staff уровне — это не "найти баг", это **передача знаний, выравнивание архитектуры и ментальных моделей** команды. ADR (Architecture Decision Record) — документирование ключевых технических решений для будущего.

---

## 🔬 Глубокий разбор (Senior/Staff)

### Принципы Staff-уровня Code Review

```
❌ Junior CR: "Тут переменная названа плохо"
❌ Mid CR: "Этот алгоритм O(n²), можно O(n log n)"
✅ Senior CR: "Это правильно для текущего масштаба, но при 10x нагрузке 
              нам понадобится batch processing. Добавь TODO с объяснением."
✅ Staff CR: "Этот подход предполагает, что сервис A знает о существовании 
             сервиса B. Это создаёт coupling. Рассмотри Event-driven подход."
```

### Что проверять на CR (чеклист Senior/Staff)

```markdown
## Correctness
- [ ] Edge cases: empty input, nil, overflow, concurrent access
- [ ] Error handling: все ошибки обработаны, не проглочены молча
- [ ] Transactions: атомарность, rollback при ошибке

## Architecture
- [ ] Single Responsibility: функция/пакет делает одно
- [ ] Coupling: нет прямых зависимостей между несвязанными доменами
- [ ] Testability: можно замокать зависимости?

## Performance
- [ ] Аллокации в hot path: создаём ли объекты в цикле?
- [ ] N+1 queries: SELECT в цикле вместо JOIN?
- [ ] Connection pools: закрываем resp.Body, rows.Close()?

## Security
- [ ] SQL injection: параметризованные запросы?
- [ ] Секреты в логах: нет ли утечки PII?
- [ ] Input validation: санитизация всех внешних данных?

## Operational
- [ ] Structured logging с correlation_id?
- [ ] Метрики для ключевых операций?
- [ ] Graceful shutdown при deployment?
```

### Тон и культура CR

```markdown
❌ "Это неправильно. Нужно использовать sync.Map."
✅ "Любопытно — в чём причина выбора map + mutex вместо sync.Map? 
    sync.Map оптимизирован для случая many reads/few writes. 
    Если у нас именно такой сценарий — стоит рассмотреть."

❌ "Эта функция слишком длинная."
✅ "Эта функция делает несколько вещей: валидацию и сохранение. 
    Как ты смотришь на разделение? Это улучшит тестируемость валидации отдельно."
```

**Принцип**: CR — это диалог, не суд. Вопросы лучше утверждений. Объясняй "почему", а не только "что".

### Architecture Decision Records (ADR)

ADR — короткий документ (~1 страница) фиксирующий: контекст решения, рассмотренные альтернативы, принятое решение и его последствия.

```markdown
# ADR-042: Использование Kafka для async заказов

**Дата**: 2024-03-15  
**Статус**: Accepted  
**Авторы**: @ivanov, @petrov

## Контекст

Синхронная обработка заказов создаёт spike'и нагрузки на payment-service 
в часы пик (до 10x baseline). Timeout'ы начали нарушать SLO.

## Рассмотренные альтернативы

| Вариант | Pros | Cons |
|---------|------|------|
| Direct HTTP | Простота | Coupling, cascading failures |
| RabbitMQ | Знакомо команде | Сложнее horizontal scaling |
| **Kafka** | **Retention, replay, partitioning** | **Операционная сложность** |
| DB polling | Простота, ACID | Latency, DB нагрузка |

## Решение

Используем Kafka для async обработки заказов.  
Outbox pattern для at-least-once delivery из БД.

## Последствия

✅ Payment-service декоuplед от order-service  
✅ Можем replay события при сбоях  
❌ Операционная сложность (Kafka кластер)  
❌ Eventual consistency — UI должен показывать "в обработке"

## Revisit условия

Если команда < 5 человек или трафик < 1000 rps — рассмотреть PostgreSQL LISTEN/NOTIFY.
```

### Хранение ADR в репозитории

```
/docs/
  architecture/
    decisions/
      ADR-001-database-choice.md
      ADR-002-api-versioning.md
      ADR-042-kafka-async-orders.md
    diagrams/
      system-context.png
```

**Принцип**: ADR — часть кода. PR с архитектурным изменением должен включать ADR. Это позволяет через год ответить на вопрос "почему мы так сделали?"

---

## 🔥 Реальные боевые кейсы

- **N+1 на CR**: код выглядел правильно, но в hot path — SELECT в цикле. Нашли через EXPLAIN в review, не в production
- **ADR сохранил команду**: новый разработчик предложил убрать Saga — ADR объяснил почему она появилась (проблема с distributed transactions)
- **CR культура**: компания перешла от "найди баги" к "обучающий review" — скорость онбординга выросла в 2x
- **Nit comments**: префикс `nit:` для стилистических замечаний — автор знает что это необязательно фиксить

## 💬 Как отвечать на интервью

> "Code Review для меня — инструмент выравнивания понимания архитектуры в команде. На Staff уровне смотрю прежде всего на coupling, testability, operational readiness (metrics, graceful shutdown), а не на форматирование — это работа линтера. ADR — обязательная практика для любого нетривиального архитектурного решения. Документ пишется когда принимается решение, не постфактум, иначе ценность теряется."

## ❓ Вопросы для интервью (Senior/Staff)

### Как справиться с тем, что команда игнорирует CR-комментарии?

Это проблема культуры, не процесса. Подход: 1) Обсудить конкретный случай 1:1. 2) Разделить "blocker" (должен исправить) от "suggestion" (желательно). 3) Автоматизировать format/lint — это убирает субъективные дискуссии. 4) Возможно, CR слишком поздно — смотреть на design review до кодирования.

### Когда ADR устаревает и что с ним делать?

Когда контекст изменился (новые требования, другая команда, технология устарела). Не удаляй старые ADR — меняй статус на `Superseded by ADR-NNN`. Это сохраняет историю "почему мы так думали тогда".

### Как CR работает в команде с высоким velocity (много PR в день)?

Small PRs wins: PR < 400 строк, один logical change. CR < 24h SLO. Пары: critical paths — 2 reviewer, прочее — 1. Feature flags позволяют merge incomplete features без блокировки. Draft PRs для early feedback перед финишем.

## 📊 Итоговая шпаргалка

| Аспект | Senior/Staff фокус |
|--------|-------------------|
| CR цель | Архитектура + знания, не только баги |
| Тон | Вопросы, а не утверждения |
| Blocker vs Nit | Явно раздели важность замечаний |
| ADR формат | Контекст → Альтернативы → Решение → Последствия |
| ADR хранение | В репо `/docs/decisions/` как код |
| Статусы ADR | Proposed → Accepted → Superseded / Deprecated |
