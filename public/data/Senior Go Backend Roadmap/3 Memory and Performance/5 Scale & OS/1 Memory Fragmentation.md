# 1. Memory Fragmentation: Mechanics

## Виды фрагментации
1. **Внутренняя**: Когда мы выделяем блок (size class) 32 байта, а объекту нужно всего 17. Остаток пропадает.
2. **Внешняя**: Когда между занятыми блоками есть свободные дырки, но ни одна из них не достаточна для нового большого объекта.

## Как Go минимизирует фрагментацию
Go использует аллокатор на базе сегментов и классов размеров (Size Classes). Вместо произвольных кусков памяти он нарезает страницы на одинаковые ячейки (например, только по 8, 16, 32, 48 ... байт). Это сводит внешнюю фрагментацию к минимуму.

## Вопросы для интервью (Senior/Staff)

### Как структуры с разным временем жизни влияют на фрагментацию?
Если вы храните долгоживущую структуру (кеш) и рядом с ней в той же странице памяти постоянно создаете и удаляете мелкие временные объекты, страница никогда не освободится полностью. Это может привести к постепенному раздуванию (bloat) виртуальной памяти процесса.

### В чем опасность использования больших массивов (например, [1000000]int) в куче?
Большие объекты требуют выделения непрерывного куска памяти (слайны/mspan). Найти такой кусок в фрагментированной куче сложнее, что может вызвать запуск GC чаще, чем нужно, или даже привести к OOM при наличии свободного места мелкими кусками.

### Как `debug.FreeOSMemory()` взаимодействует с ОС?
Этот вызов принудительно запускает GC и пытается вернуть свободные страницы памяти операционной системе (через `madvise(MADV_DONTNEED)`). В обычном режиме Go делает это лениво и постепенно. Вызывать это вручную в высоконагруженных системах опасно, так как последующие аллокации заставят ОС снова выделять страницы (Page Faults), что ударит по производительности.

### Какую пользу приносит Runtime Metrics API для отслеживания фрагментации?
Из `runtime/metrics` можно получить точные данные о количестве свободных и занятых байт в каждом "классе размера" (Size Class). Это позволяет понять, не тратим ли мы 40% памяти просто на паддинг и пустые остатки в блоках.
