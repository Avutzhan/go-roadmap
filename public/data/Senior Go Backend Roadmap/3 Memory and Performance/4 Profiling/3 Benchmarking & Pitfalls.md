# 3. Benchmarking & Pitfalls

## Написание бенчмарков в Go
Бенчмарки пишутся в портативных `_test.go` файлах и запускаются через `go test -bench`.

### Основные правила
- Использовать цикл `for i := 0; i < b.N; i++`. `b.N` подбирается рантаймом автоматически.
- `b.ResetTimer()`: если перед тестом идет долгая подготовка данных.
- `b.ReportAllocs()`: обязателен для анализа давления на GC.

## Вопросы для интервью (Senior/Staff)

### Как избежать оптимизации компилятором (Dead Code Elimination) в бенчмарке?
Если результат функции не используется, компилятор может просто выкинуть весь вызов.
**Решение**: записывать результат в глобальную переменную (пакетного уровня).
```go
var globalResult int
func BenchmarkMyFunc(b *testing.B) {
    var r int
    for i := 0; i < b.N; i++ {
        r = MyFunc()
    }
    globalResult = r
}
```

### Что такое Benchmark Parallelism (b.RunParallel)?
Позволяет запустить бенчмарк в несколько горутин параллельно (количество равно `GOMAXPROCS`). Это критично для тестирования объектов, защищенных мьютексами, чтобы увидеть влияние конкуренции (Lock Contention) на производительность.

### Почему результаты бенчмарков на ноутбуке могут быть нестабильны?
1. **Turbo Boost**: Процессор меняет частоту. Для точности его нужно "залочить" на одной частоте.
2. **Фоновые процессы**: Другие программы забирают CPU.
3. **Термальный троттлинг**: При долгих тестах процессор перегревается и снижает скорость.
**Staff совет**: используйте `benchstat` для сравнения нескольких прогонов и проверки статистической значимости улучшений.

### В чем разница между Benchmarking и Fuzzing?
Бенчмарки проверяют **скорость** на известных данных. Физзинг (Fuzzing) проверяет **корректность**, подавая на вход случайные мусорные данные, чтобы найти edge-cases и паники. С Go 1.18 фаззинг встроен в стандартный тулчейн.
