# 3. Compiler Optimizations (BCE, Inlining)

## Инлайнинг (Inlining)
Процесс замены вызова функции самим телом этой функции.
- **Плюс**: Убирает оверхед на передачу аргументов в стек и переход. Позволяет компилятору лучше проводить Escape Analysis (локальные переменные функции больше не "убегают" при передаче во внутренние вызовы).
- **Минус**: Увеличивает размер бинарного файла.
Компилятор Go инлайнит только простые функции (бюджет инлайнинга).

## Bound Check Elimination (BCE)
При каждом обращении к элементу слайса `s[i]` Go вставляет проверку границ в рантайме. Если `i >= len(s)`, программа упадет с паникой.
BCE — это оптимизация, при которой компилятор убирает эти проверки, если может статически доказать безопасность доступа.

## Вопросы для интервью (Senior/Staff)

### Как форсировать или запретить инлайнинг?
- `//go:noinline`: запрещает инлайнинг конкретной функции.
- Для форсирования явного указания нет, но можно упрощать функцию, разбивая её на мелкие части.
Проверить инлайнинг можно через `go build -gcflags="-m"`.

### Пример хака для BCE (Bound Check Elimination)
Если вы работаете со слайсом в цикле, вы можете в начале функции сделать одну явную проверку: `_ = s[len(s)-1]`. После этого компилятор поймет, что все индексы внутри цикла безопасны, и уберет проверки для каждой итерации. На миллионах итераций это дает значительный прирост.

### Что такое Devirtualization?
Это когда компилятор заменяет вызов метода через интерфейс прямым вызовом конкретной функции (если он может точно определить тип объекта на этапе сборки). Это сокращает путь выполнения и открывает двери для инлайнинга.

### Как `escape analysis` взаимодействует с `inlining`?
Если функция не заинлайнена, все ссылки, передаваемые в нее, с большой вероятностью "убегут" в кучу. Если функция заинлайнена, компилятор видит контекст использования переменных и может оставить их на стеке. Это одна из важнейших причин, почему маленькие функции в Go очень эффективны.
