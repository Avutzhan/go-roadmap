# 2. Alignment & Padding

## Выравнивание памяти (Memory Alignment)
Процессоры читают память не по байтам, а "словами" (обычно 8 байт на 64-битных системах). Чтобы чтение было эффективным за один такт, данные должны быть выровнены.

### Паддинг (Padding)
Если поля структуры стоят в "неудачном" порядке, компилятор вставит пустые байты (отступы) между ними.
```go
type Example struct {
    a int8  // 1 байт
    // 7 байт отступа (padding)
    b int64 // 8 байт
    c int8  // 1 байт
    // 7 байт отступа (padding)
} // Итого: 24 байта вместо 10 полезных
```

## Оптимизация (Порядок полей)
Для минимизации размера структуры поля следует располагать **от большего к меньшему** (по убыванию размера типа).

## Вопросы для интервью (Senior/Staff)

### Как проверить размер структуры и выравнивание программно?
Использовать методы пакета `unsafe`: `unsafe.Sizeof()`, `unsafe.Alignof()` и `unsafe.Offsetof()`. Это позволяет точно понять, во что превращается ваша структура в памяти.

### Влияет ли порядок полей на производительность?
Да, напрямую. 
1. **Размер**: Меньше структура — больше их влезает в одну кэш-линию процессора.
2. **False Sharing**: Если две горутины часто пишут в соседние поля одной структуры, они инвалидируют кэш друг друга. Иногда специально добавляют паддинг, чтобы разнести поля в разные кэш-линии.

### Что такое "пустая структура в конце" (Empty struct at the end of struct)?
Если поле `struct{}` является последним в структуре, оно может увеличить размер всей структуры. Это происходит потому, что рантайму нужен уникальный адрес для этого поля, и он добавляет 1 или 8 байт паддинга, чтобы адрес не указывал за пределы выделенного блока (иначе GC может сойти с ума).
**Staff совет**: переместите пустые структуры в начало.

### Как инструменты типа `fieldalignment` помогают в разработке?
Это линтеры (входят в состав `golangci-lint`), которые автоматически находят неоптимальные структуры и предлагают их переупорядочить. Для высоконагруженных систем использование таких проверок обязательно.
