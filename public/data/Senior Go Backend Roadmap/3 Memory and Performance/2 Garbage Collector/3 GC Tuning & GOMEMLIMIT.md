# 3. GC Tuning & GOMEMLIMIT

## Управление поведением GC
В Go нет сотен флагов тюнинга, как в JVM. Есть два основных рычага.

### GOGC (Процент роста кучи)
По умолчанию `100`. Это значит, что новый цикл GC запустится, когда размер кучи вырастет на 100% относительно размера после предыдущей очистки.
- `GOGC=off`: полностью отключает GC.
- `GOGC=50`: запускать в два раза чаще (меньше памяти, больше CPU).
- `GOGC=500`: запускать реже (больше памяти, меньше CPU).

### GOMEMLIMIT (Go 1.19+)
Это "мягкий" лимит памяти. Он указывает рантайму: "старайся не превышать этот объем памяти любой ценой". При подходе к лимиту GC начинает работать более агрессивно, игнорируя настройки `GOGC`.

## Вопросы для интервью (Senior/Staff)

### Как GOMEMLIMIT решает проблему OOM (Out of Memory) в Kubernetes?
Раньше, если у вас был лимит контейнера 1GB, а `GOGC=100`, GC мог не запуститься, пока память не достигнет 1.2GB (если после прошлого цикла было 600MB). Приложение убивалось по OOM Killer. Теперь установка `GOMEMLIMIT=900MiB` гарантирует, что GC сработает вовремя, предотвращая перезагрузку сервиса.

### В чем опасность Ballast (балластного слайса) и нужен ли он сейчас?
Балласт — это огромный неиспользуемый слайс байт, выделяемый на старте, чтобы "обмануть" расчет `GOGC` и сделать циклы реже. С появлением `GOMEMLIMIT` этот хак больше не нужен и считается антипаттерном.

### Что происходит, если GOMEMLIMIT установлен слишком низко?
Приложение попадет в "GC Death Spiral" — рантайм будет тратить 100% CPU на постоянную очистку памяти, пытаясь уложиться в лимит, вместо выполнения бизнес-логики. Staff-инженер должен уметь ловить это состояние через метрики `GC CPU Fraction`.

### Какие метрики критичны для мониторинга тюнинга GC?
1. `go_gc_duration_seconds`: задержки.
2. `go_memstats_heap_alloc_bytes`: размер живого графа.
3. `go_gc_pause_total_ns`: накопленное STW время.
4. `go_gc_cpu_fraction`: процент ресурсов CPU, отнимаемых Garbage Collector'ом.
