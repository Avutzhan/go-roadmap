# 2. Write Barriers & Latency

## Механика Write Barrier
В Go используется гибридный барьер записи (Hybrid Write Barrier). Он гарантирует, что никакие объекты не будут потеряны в процессе конкурентной разметки, пока "мутатор" (ваша программа) меняет ссылки.

### Как это влияет на производительность?
Когда GC активен, каждая запись указателя в памяти обходится дороже на несколько процессорных инструкций. Это незаметно в обычном коде, но критично в циклах с интенсивной манипуляцией ссылками.

## Latency (Задержки)
Go GC оптимизирован так, чтобы паузы Stop-The-World (STW) не превышали 1 миллисекунду.
- **Штатные STW**: Происходят в начале (подготовка барьеров) и в конце (завершение разметки) цикла.
- **Проблемные STW**: Если планировщик не может быстро остановить горутины в безопасной точке (Safe Point).

## Вопросы для интервью (Senior/Staff)

### Что такое Safe Points?
Это точки в коде, где рантайм может безопасно приостановить выполнение горутины для нужд GC. Обычно это вызовы функций. Если горутина застряла в длинном цикле без вызовов функций (и система асинхронного вытеснения Go 1.14+ не сработала), она может задерживать начало фазы STW, увеличивая общую задержку системы.

### В чем разница между Throughput и Latency в контексте GC?
- **Throughput** (пропускная способность): Сколько всего времени CPU тратит на полезную работу. GC с полной остановкой (как в старой Java) имеет высокий throughput, но ужасную latency.
- **Latency** (задержка): Время одиночной паузы. Go GC выбирает низкую задержку, жертвуя общим throughput (тратит больше CPU на частые циклы).

### Как бороться с "GC Thrashing"?
Это ситуация, когда GC запускается слишком часто (почти постоянно). Причина — огромный темп аллокаций короткоживущих объектов. Решение: пулы объектов (`sync.Pool`), оптимизация структур (выравнивание) и уменьшение количества "побегов" в кучу.

### Влияние системных вызовов (syscalls) на задержки GC
Горутины в системных вызовах не блокируют GC. Рантайм считает их "вне игры" и сканирует их стеки без остановки потока M. Это позволяет приложениям с интенсивным вводом-выводом сохранять низкую задержку даже при больших объемах данных.
