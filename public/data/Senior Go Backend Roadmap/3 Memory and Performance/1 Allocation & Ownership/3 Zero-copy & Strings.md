# 3. Zero-copy & Strings

## Оптимизация обработки строк и байт
В Go `string` и `[]byte` имеют разную структуру: слайс содержит указатель на данные, длину и емкость, а строка — только указатель и длину. Также строки неизменяемы.

### Проблема копирования
Классическое преобразование `string([]byte)` вызывает полное копирование данных, чтобы гарантировать неизменяемость строки. На больших данных это создает двойную нагрузку на память и GC.

## Техника Zero-copy (Go 1.20+)
С версии 1.20 появились официальные методы в пакете `unsafe` для преобразования без копирования:
```go
// []byte -> string
s := unsafe.String(&b[0], len(b))

// string -> []byte
b := unsafe.Slice(unsafe.StringData(s), len(s))
```
Это позволяет работать со строками так же быстро, как с байтами, избегая аллокаций.

## Вопросы для интервью (Senior/Staff)

### В чем риск использования unsafe для преобразования строк?
Главный риск — неизменяемость. Если вы превратили `[]byte` в `string` без копирования, а затем изменили исходный слайс байт, значение строки тоже изменится. Это нарушает гарантии языка Go и может привести к непредсказуемым багам, если строка используется как ключ в мапе или передается в библиотеки, ожидающие константности.

### Почему strings.Builder эффективнее конкатенации через +?
Конкатенация строк через `+` каждый раз создает новую строку и копирует в нее данные старых. `strings.Builder` использует внутренний слайс байт с геометрическим ростом емкости (как под капотом у слайсов), что сводит количество аллокаций к логарифмическому минимуму.

### Что такое String Interning и поддерживает ли его Go?
Интернирование — это хранение только одного экземпляра одинаковых строк для экономии памяти. Go автоматически интернирует только строковые литералы (константы в коде). Для динамических строк из сети/БД автоматического интернирования нет, но Staff-инженер может реализовать его вручную через мапу или использовать сторонние библиотеки для дедупликации строк в памяти.

### Как работает подстрока (substring) в Go и может ли она вызвать утечку памяти?
Подстрока `s2 := s1[1:5]` не копирует данные, а просто создает новый заголовок `string`, указывающий на тот же массив байт в куче. Если `s1` — это огромный текст (100MB), а `s2` — маленькое слово из него, то весь массив 100MB не будет очищен GC, пока жива ссылка `s2`. Это классическая утечка памяти. Решение: `s2 := string([]byte(s1[1:5]))` для форсированного копирования.
