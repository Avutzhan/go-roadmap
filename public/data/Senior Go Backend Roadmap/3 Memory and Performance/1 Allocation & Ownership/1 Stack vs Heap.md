# 1. Stack vs Heap: Memory Layout

## Разделение памяти в рантайме Go
В Go память делится на **Stack** (стек) и **Heap** (куча). Понимание того, где окажутся ваши данные — это ключ к производительности уровня Staff.

### Стек (Stack)
- **Локальность**: Каждая горутина имеет свой стек (начальный размер 2KB).
- **Цена**: Аллокация на по принципу LIFO (Last-In-First-Out) — это просто изменение значения регистра SP (Stack Pointer). Это практически бесплатно.
- **Освобождение**: Происходит автоматически при выходе из функции. Никакой нагрузки на Garbage Collector.

### Куча (Heap)
- **Общий ресурс**: Глобальное пространство памяти, доступное всем горутинам.
- **Цена**: Аллокация требует поиска свободного блока через `mcache`, `mcentral` или `mheap`. Это дорого и может вызвать конкуренцию за мьютексы в рантайме.
- **Освобождение**: Память очищается Garbage Collector'ом через циклы разметки. Это создает CPU оверхед и вносит задержки (STW).

## Вопросы для интервью (Senior/Staff)

### В каких случаях стек горутины копируется и как это влияет на производительность?
Стек копируется, когда он переполняется (больше 2KB и далее в 2 раза). Рантайм выделяет новый сегмент и копирует туда данные. Это вносит небольшую задержку, но критично то, что рантайм должен обновить все указатели внутри горутины, указывающие на переменные в этом стеке. Для Staff-инженера важно знать, что "бесконечная" глубокая рекурсия может вызвать сотни таких копирований, что приведет к деградации.

### Роль mcache в аллокации памяти Go
`mcache` — это структура данных, привязанная к каждому логическому процессору **P**. Она содержит списки свободных блоков памяти разных размеров (size classes). Это позволяет горутине выделять память в куче без блокировок мьютексом, пока в `mcache` есть свободные блоки. Это критическая оптимизация для параллелизма (основана на алгоритме TCMalloc).

### Почему структуры на стеке лучше для кэша процессора?
Данные на стеке обычно плотно упакованы и чаще находятся в кэше процессора (L1/L2). Данные в куче могут быть разбросаны по разным страницам памяти, что приводит к частым промахам кэша (cache misses) и падению производительности даже без учета GC.
