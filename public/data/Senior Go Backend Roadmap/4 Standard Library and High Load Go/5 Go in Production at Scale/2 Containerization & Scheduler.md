# 2. Containerization & Scheduler (CFS Throttling)

## Go и Docker CPU Limits
Go рантайм по умолчанию смотрит на количество ядер всей физической машины, а не на лимиты контейнера.
- **Проблема**: Если на ноде 64 ядра, а у контейнера лимит `cpu: 2`, Go создаст 64 процессора **P**. Планировщик будет пытаться загрузить все 64 P, но ядро Linux (через CFS quota) будет жестко прерывать процесс, увидев превышение лимита в 2 ядра.
- **Результат**: Огромные задержки (Latency Spikes), так как горутины будут "замерзать" на середине выполнения.

## Библиотека `automaxprocs` (от Uber)
Она автоматически считывает лимиты из cgroups и устанавливает `GOMAXPROCS` в корректное значение. Использование этой библиотеки в `main.go` обязательно для всех Go сервисов в Kubernetes.

## Вопросы для интервью (Senior/Staff)

### В чем разница между CPU Requests и CPU Limits в контексте Go планировщика?
- **Requests**: Влияет на приоритезацию (сколько CPU приложению гарантированно дадут).
- **Limits**: Жесткая граница. Если планировщик Go создаст больше потоков M, чем разрешено лимитом, процесс будет задушен (throttled).

### Как OOM Killer выбирает жертву (Go сервис против других)?
Система смотрит на `oom_score`, который зависит от процента используемой памяти от лимита контейнера. Если ваш Go сервис сожрал 95% памяти без `GOMEMLIMIT`, он будет первой целью.

### Почему важно отключать Swapping для Go приложений?
Go рантайм и Garbage Collector рассчитывают на быстрый доступ к памяти. Если страницы памяти уйдут на диск (swap), сборка мусора превратится в катастрофу, так как сканирование кучи будет вызывать постоянные I/O операции.

### Какую пользу приносит Sidecar Container для профилирования?
Позволяет снимать дампы памяти и CPU без прямого вмешательства в код приложения и перезапуска. Это часть философии Cloud Native Observability.
