# 2. Keep-alive & Middleware Patterns

## HTTP Keep-Alive
Механизм, позволяющий использовать одно TCP-соединение для отправки нескольких HTTP запросов.
- **Преимущество**: Экономия на трехстороннем рукопожатии TCP (Handshake) и TLS-negotiation.
- **Нюанс**: Если вы прочитали не все тело ответа (`io.Copy(io.Discard, resp.Body)`), соединение может быть закрыто вместо возврата в пул.

## Middleware Patterns
В Go middleware — это просто функции, которые принимают `http.Handler` и возвращают `http.Handler`.

### Цепочки (Chaining)
Паттерн декоратора позволяет прозрачно добавлять функционал:
1. Логирование.
2. Трейсинг (Trace-ID).
3. Аутентификация.
4. Recovery (обработка паник).

## Вопросы для интервью (Senior/Staff)

### Как реализовать Middleware, которое умеет изменять статус-код ответа?
Нужно создать обертку над `http.ResponseWriter` (структуру-шпиона), которая перехватывает вызов `WriteHeader(int)`. Это стандартный подход для логирования кодов ошибок в middleware.

### В чем разница между `http.Handler` и `http.HandlerFunc`?
- `http.Handler` — это интерфейс с методом `ServeHTTP`.
- `http.HandlerFunc` — это тип-функция, которая реализует этот интерфейс. Это синтаксический сахар, позволяющий использовать обычные функции как хендлеры.

### Как работает IdleConnTimeout и почему он важен?
Это время, в течение которого неиспользуемое соединение может лежать в пуле. Если установить слишком большое значение, вы можете держать тысячи "полуживых" соединений со стороны балансировщика или сервера, что нагружает память и лимиты ОС. Если слишком маленькое — вы теряете преимущество Keep-Alive.

### Проблема "Thundering Herd" при рестарте сервиса с Keep-Alive
При массовом переподключении тысяч клиентов после деплоя может произойти всплеск нагрузки на CPU из-за TLS-хендшейков. Решение: постепенный прогрев (warm-up) или использование интеллектуальных балансировщиков нагрузки (L7).
