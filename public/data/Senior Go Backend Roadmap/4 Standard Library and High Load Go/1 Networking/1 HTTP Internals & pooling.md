# 1. HTTP Internals & Connection Pooling

## Устройство net/http
В Go `net/http` — это мощный, промышленный пакет, но его неправильная настройка на High-load может привести к отказу сервиса.

### Клиент: http.Client & http.Transport
`http.Client` — это высокоуровневая обертка. Вся магия происходит в `http.Transport`.
- **RoundTripper**: Интерфейс, который выполняет один HTTP запрос.
- **Connection Pooling**: `Transport` кэширует TCP-соединения для повторного использования (Keep-Alive).

## Тонкая настройка пула (Connection Pooling)
По умолчанию настройки Go `DefaultTransport` слишком консервативны для микросервисов:
- `MaxIdleConns`: Всего 100 на весь транспорт.
- `MaxIdleConnsPerHost`: Всего 2 (!!) на один хост.
**Проблема**: При высокой нагрузке на один сервис новые запросы будут постоянно открывать и закрывать TCP-соединения, уходя в состояние `TIME_WAIT` и исчерпывая порты ОС.

**Staff рекомендация**:
```go
transport := &http.Transport{
    MaxIdleConns:        1000,
    MaxIdleConnsPerHost: 100,
    IdleConnTimeout:     90 * time.Second,
}
```

## Вопросы для интервью (Senior/Staff)

### Почему важно всегда закрывать `resp.Body`?
Если тело ответа не закрыто, TCP-соединение не может быть возвращено в пул для повторного использования. Оно останется "занятым" до момента, пока сборщик мусора не очистит объект, что приведет к утечке дескрипторов и ресурсов. Правильный паттерн: `defer resp.Body.Close()` сразу после проверки на ошибку.

### Как работает `DefaultClient` и в чем его опасность?
`http.DefaultClient` использует `http.DefaultTransport`, у которого **нет таймаутов** по умолчанию. Если внешний сервис "зависнет" и не будет отдавать данные, ваша горутина будет висеть вечно, потребляя память. В продакшене **всегда** создавайте свой `http.Client` с явным `Timeout`.

### Что такое `Hijacking` в HTTP сервере?
Это механизм (`http.Hijacker`), позволяющий обработчику запроса перехватить управление TCP-соединением у HTTP-сервера. После вызова `Hijack()` сервер больше не управляет этим соединением, и вы можете писать в сокет сырые данные (используется для WebSockets или апгрейда до других протоколов).

### Влияние `GOMAXPROCS` на производительность HTTP сервера
HTTP сервер в Go создает новую горутину на каждое входящее соединение. Планировщик эффективно распределяет их по P. На High-load важно следить, чтобы количество P соответствовало лимитам CPU в контейнере, иначе конкуренция за ядра (throttling) приведет к резкому росту Latency (P99).
