# 2. JSON & Performance: Reflection vs Codegen

## Стандартный `encoding/json`
Работает на базе рефлексии. При каждом маршалинге Go должен просканировать структуру, найти теги и сопоставить поля.
**Staff уровень**: На High-load это становится бутылочным горлышком из-за CPU оверхеда и аллокаций в куче.

## Альтернативы (Кодогенерация)
Библиотеки типа `easyjson` или `ffjson` генерируют код сериализации на этапе сборки.
- **Плюс**: В 2-5 раз быстрее за счет отсутствия рефлексии.
- **Минус**: Усложнение CI/CD (нужно генерировать файлы) и увеличение размера бинарного файла.

## Вопросы для интервью (Senior/Staff)

### Как обработать JSON размером 10GB в Go?
Использовать `json.Decoder`. Вместо `json.Unmarshal` (который требует весь слайс в памяти), декодер читает поток и позволяет декодировать объекты по одному (например, элементы массива) в цикле. Память остается на уровне размера одного объекта.

### В чем опасность использования `interface{}` (any) в JSON структурах?
При декодировании `encoding/json` превращает числа в `float64`, а мапы — в `map[string]interface{}`. На больших данных это создает огромное количество аллокаций. Старайтесь использовать строго типизированные структуры везде, где это возможно.

### Как `json.RawMessage` помогает оптимизировать парсинг?
Он позволяет отложить парсинг части JSON. Вы можете прочитать только метаданные, а тяжелую "полезную нагрузку" оставить в виде сырых байт (`[]byte`) и распарсить её позже или передать другому сервису без лишней десериализации/сериализации.

### Существуют ли быстрые альтернативы JSON без кодогенерации?
Да, библиотеки типа `json-iterator` стараются оптимизировать рефлексию через кэширование структур, но они всё равно обычно уступают кодогенерации или бинарным протоколам (Protobuf, MsgPack).
