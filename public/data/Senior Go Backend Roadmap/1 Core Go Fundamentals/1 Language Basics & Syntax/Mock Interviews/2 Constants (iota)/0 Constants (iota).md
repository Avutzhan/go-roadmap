# 🧠 2. Constants & iota

## 📌 Что это такое

`const` в Go — это значения, вычисляемые **во время компиляции**. Это принципиально важно: они хранятся не в heap и не в stack, а встраиваются прямо в бинарник.

`iota` — это специальный счётчик, который сбрасывается в 0 в начале каждого блока `const` и инкрементируется при каждой NEW строке константы.

```go
const (
    A = iota // 0
    B         // 1
    C         // 2
)
```

## 🔬 Глубокий разбор (Senior/Staff)

### Типизированные vs нетипизированные константы

В Go константы могут быть **нетипизированными** (untyped). Это критически важная деталь.

```go
const UntypedMax = 1000        // нетипизированная — "идеальное число"
const TypedMax int32 = 1000    // типизированная — строго int32

func takeInt64(n int64) {}

takeInt64(UntypedMax) // ✅ работает — компилятор адаптирует тип
takeInt64(TypedMax)   // ❌ compile error: cannot use int32 as int64
```

**Нетипизированная константа** имеет "идеальный" тип и автоматически приводится к нужному числовому типу в точке использования. Это основа гибкости числовых литералов в Go.

### iota: битовые флаги (классический паттерн)

Самый частый реальный кейс — задание permissions через bitwise операции:

```go
type Permission uint8

const (
    Read    Permission = 1 << iota // 1 << 0 = 1  (00000001)
    Write                          // 1 << 1 = 2  (00000010)
    Execute                        // 1 << 2 = 4  (00000100)
)

func hasPermission(userPerms, required Permission) bool {
    return userPerms&required != 0
}

// Использование
perms := Read | Write          // 00000011
fmt.Println(hasPermission(perms, Execute)) // false
fmt.Println(hasPermission(perms, Write))   // true
```

### iota: пропуски и выражения

```go
const (
    _           = iota             // 0 — игнорируем нулевое значение
    KB ByteSize = 1 << (10 * iota) // 1 << 10 = 1024
    MB                             // 1 << 20
    GB                             // 1 << 30
    TB                             // 1 << 40
)
```

**Правило**: в каждой строке `iota` берёт **своё значение** (0, 1, 2...), но выражение применяется повторно с новым `iota`.

### const vs var: разница в памяти

```go
// const — встроен в бинарник, не адресуем
const MaxRetries = 3
// &MaxRetries // ❌ compile error: cannot take address of const

// var — находится в памяти, адресуем
var MaxConnections = 100
// &MaxConnections // ✅ OK
```

Это значит: **нельзя передать константу по указателю**. Для настроек, которые надо передавать в функции, используй `var` или параметры.

## 🔥 Реальные боевые кейсы

- **HTTP Status codes** — константы для статусов (`http.StatusOK = 200`)
- **Enum-like типы** через iota + `String()` метод для логирования
- **Bit flags** для прав доступа (RBAC), состояний (FSM)
- **Size constants** для буферов (без магических чисел)

```go
// Anti-pattern: магические числа
if status == 3 { ... }

// Best practice: именованные константы + iota
type OrderStatus int
const (
    OrderPending   OrderStatus = iota + 1 // 1
    OrderPaid                             // 2
    OrderShipped                          // 3
    OrderDelivered                        // 4
)

func (s OrderStatus) String() string {
    return [...]string{"", "Pending", "Paid", "Shipped", "Delivered"}[s]
}
```

Нетипизированные константы имеют 'идеальный' тип и автоматически приводятся в точке использования — это даёт гибкость без явного приведения типов. `iota` я использую для битовых флагов, enum-подобных типов и мультипликативных размерных констант. ## ❓ Вопросы для интервью (Senior/Staff)

### В чём опасность typed iota enum без явного `String()` метода?

Если добавить новое значение в середину iota-блока, все последующие значения сдвинутся. Если эти значения хранятся в БД (как int), то после миграции данные станут некорректными. **Решение**: никогда не добавлять значения в середину — только в конец, или использовать явные числовые значения вместо iota для персистентных enum.

### Почему `const` не может быть типа `slice` или `map`?

Потому что `const` должен быть вычислим полностью во время компиляции, а slice и map — это runtime-структуры с динамической памятью и GC. Для readonly-коллекций используй `var` с соглашением "не модифицировать" или функции, возвращающие новый slice каждый раз.

### Как создать thread-safe enum с проверкой диапазона?

```go
type Status int

const (
    StatusActive Status = iota + 1
    StatusInactive
    statusSentinel // неэкспортируемый "страж"
)

func (s Status) IsValid() bool {
    return s > 0 && s < statusSentinel
}
```

Используй неэкспортируемую `sentinelValue` как верхнюю границу. Это позволяет валидировать enum без хардкода числа.

### Чем отличается `iota` в разных блоках `const`?

`iota` **сбрасывается в 0** при каждом новом ключевом слове `const`. Два отдельных блока — два независимых счётчика.

```go
const A = iota // 0 (первый блок)
const B = iota // 0 (второй блок — сброс!)
```

## 📊 Итоговая шпаргалка

| Аспект | Детали |
|--------|--------|
| Место хранения | Бинарник (rodata) — нет GC оверхеда |
| Адресуемость | ❌ Нельзя взять адрес (`&const`) |
| Нетипизированные | Авто-конвертируются в точке использования |
| iota | Сброс в 0 при каждом новом `const` блоке |
| Bit flags | `1 << iota` — классика для permissions/FSM |
| Enum в БД | Не доверяй iota-порядку — используй явные значения |
