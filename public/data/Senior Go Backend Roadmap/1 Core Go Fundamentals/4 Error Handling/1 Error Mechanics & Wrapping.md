# 1. Error Mechanics & Wrapping

## Интерфейс error
В Go ошибка — это интерфейс (builtin.go):
```go
type error interface {
    Error() string
}
```

## Wrapping как контракт API
Оборачивание ошибки через `%w` делает внутреннюю ошибку частью публичного контракта вашего API. Это означает:
- Вы разрешаете пользователю использовать `errors.Is` и `errors.As` для проверки исходной причины.
- Вы берете на себя обязательство сохранять этот тип ошибки в будущем, чтобы не сломать код пользователя.
**Правило:** Оборачивайте ошибку только если хотите предоставить пользователю возможность программно реагировать на конкретную причину. В остальных случаях используйте `%v`.

---

## Проверка ошибок: Is vs ==
- **Оператор `==`**: Проверяет только точное соответствие значения ошибки (sentinel check). Работает только если ошибка не была обернута.
- **Метод `errors.Is`**: Рекурсивно проходит по всей цепочке вложенности (Unwrap), проверяя каждый уровень на соответствие. Это стандарт для современного Go.

## Вопросы для интервью (Senior/Staff)

### Почему глубокие цепочки Wrapping — это плохо?
Глубокое вложение (более 3-4 уровней) усложняет отладку и делает логи нечитаемыми. Это также создает избыточность контекста. Рекомендуется добавлять контекст только на значимых границах логики.

### Как работает объединение ошибок (Go 1.20+)?
`errors.Join` и `%w` с несколькими аргументами позволяют создать древовидную структуру ошибок. `errors.Is` и `errors.As` будут корректно обходить все ветви этого дерева в поисках совпадения.

### В чем разница между sentinel error и типом ошибки?
- Sentinel error (переменная) проверяется через `errors.Is`.
- Тип ошибки (структура) проверяется через `errors.As`, что позволяет извлечь из нее дополнительные данные.
