# 2. Custom Errors & Strategies

## Стратегия: Log OR Return
Это фундаментальное правило чистого кода в Go:
- **Никогда не логируйте и не возвращайте ошибку одновременно.**
- Это ведет к дублированию логов (ошибка будет залогирована на каждом уровне стека), что засоряет систему мониторинга и затрудняет поиск реальной причины.
- Логировать ошибку должен тот уровень, который ее окончательно обрабатывает (например, HTTP-handler или фоновый воркер).

## Обертывание для контекста, а не для логирования
`fmt.Errorf("failed to process user %d: %w", userID, err)`
Используйте wrapping исключительно для обогащения ошибки бизнес-контекстом, который поможет в расследовании, а не для дублирования сообщения, которое уже есть в `err.Error()`.

## Вопросы для интервью (Senior/Staff)

### Почему стратегия "Accept interfaces, return structs" важна для ошибок?
Если ваша функция возвращает конкретную структуру ошибки (даже если в сигнатуре стоит `error`), вы даете вызывающему коду возможность использовать `errors.As` для получения деталей без гадания. Это делает ваш API прозрачным.

### Что такое Opaque Errors (Непрозрачные ошибки)?
Паттерн, при котором вы возвращаете ошибку, но не экспортируете ее тип. Пользователь может проверять свойства ошибки только через ваши публичные функции (например, `IsTemporary(err error) bool`). Это позволяет менять внутреннюю структуру ошибки, не ломая код пользователя.

### Когда использовать Sentinel Errors?
Только для самых общих случаев (например, `ErrNotFound`, `ErrPermissionDenied`), которые наверняка будут проверяться во многих местах приложения. Помните, что они создают жесткую зависимость между пакетами.
