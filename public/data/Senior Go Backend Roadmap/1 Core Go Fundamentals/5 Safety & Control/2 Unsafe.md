# 2. Unsafe Package: Pointer & Memory

## Суть unsafe.Pointer
`unsafe.Pointer` позволяет обходить систему типизации Go. Он может быть преобразован в любой указатель или в `uintptr` для адресной арифметики.

## Критическое правило uintptr
`uintptr` — это просто число. Garbage Collector не считает его за указатель. 
**Правило:** Нельзя хранить `uintptr` в переменной между точками безопасного завершения работы GC (safepoints). Если GC запустится и переместит объект, значение в `uintptr` станет невалидным, указывая на мусор или чужую память. Преобразование `Pointer -> uintptr -> Pointer` должно быть атомарным (выполняться в одной строке).

## Структуры и Alignment (Выравнивание)
Go выравнивает поля структур в памяти для оптимизации доступа процессора. Пакет `unsafe` часто используется для анализа этого выравнивания (например, через `unsafe.Offsetof`). Оптимизация порядка полей (от больших к меньшим) помогает уменьшить размер структуры за счет минимизации "дырок" (padding).

## Когда использование оправдано?
1. Работа с системными вызовами ОС.
2. Zero-copy конвертации (например, byte slice в string header).
3. Низкоуровневые оптимизации в рантайме или библиотеках.

## Вопросы для интервью (Senior/Staff)

### Чем опасен неправильный каст через unsafe?
Это может привести к нарушению выравнивания (misaligned access), что на некоторых архитектурах вызывает панику, а на других — серьезное падение производительности. Также это нарушает инварианты типов, которые ожидает компилятор.

### Почему использование unsafe — это риск для стабильности?
Код, полагающийся на внутреннее устройство структур рантайма или порядок байт, может сломаться при обновлении минорной версии Go, так как эти детали не являются частью публичного контракта языка.
