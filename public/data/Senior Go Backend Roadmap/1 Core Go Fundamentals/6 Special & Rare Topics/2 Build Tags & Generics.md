# 2. Build Tags & Generics

## Build Tags (//go:build)
Механизм условной компиляции. Позволяет включать или исключать файлы из сборки на основе ОС, архитектуры или тегов пользователя. Используется для разделения логики (например, разные реализации сетевого стека для Linux и Windows).

## Generics (Go 1.18+)
Дженерики позволяют писать типобезопасный универсальный код.

### Constraints (Ограничения типов)
Определяют, какие типы могут быть переданы в качестве параметров:
- **any**: любой тип (аналог `interface{}`).
- **comparable**: типы, поддерживающие операторы `==` и `!=`.
- **Custom Constraints**: интерфейсы с перечислением типов (например, `type Number interface { int | float64 }`).

### Type Inference (Вывод типов)
В большинстве случаев компилятор Go автоматически выводит типы аргументов. Однако в сложных случаях (например, когда тип используется только в возвращаемом значении) требуется явное указание: `funcName[int](args)`.

## Вопросы для интервью (Senior/Staff)

### Как дженерики влияют на производительность?
Компилятор Go использует комбинацию техник (Stenciling и Dictionary mapping). Это позволяет достичь производительности, близкой к нативной для конкретных типов, устраняя оверхед на динамическую диспетчеризацию (itab), характерный для интерфейсов.

### В чем опасность злоупотребления дженериками?
Они могут сделать код избыточно сложным и замедлить время компиляции. Дженерики следует применять для универсальных алгоритмов и структур данных, а не для сокрытия бизнес-отличий внутри доменной логики.
