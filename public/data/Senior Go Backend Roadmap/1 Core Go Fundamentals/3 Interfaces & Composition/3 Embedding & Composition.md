# 3. Embedding & Composition

## Embedding vs Inheritance
Go заменяет классическое наследование композицией через встраивание. Это устраняет проблему хрупких базовых классов (fragile base class).

### Основные механизмы:
- **Promotion**: Поля и методы встроенного типа доступны напрямую у внешнего типа.
- **Shadowing**: Внешний тип может переопределить метод встроенного. Доступ к оригиналу остается через `s.InnerType.Method()`.

## Method Sets и встраивание (Критически важно)
Взаимодействие ресиверов (Value vs Pointer) при встраивании определяет, удовлетворяет ли внешняя структура интерфейсу:

1. **Embedding by Value (`type B struct { A }`)**:
   - Если метод у `A` на `Value` ресивере -> доступен у `B` как по значению, так и по указателю.
   - Если метод у `A` на `Pointer` ресивере -> доступен у `B` только через указатель `*B`.

2. **Embedding by Pointer (`type B struct { *A }`)**:
   - Любой метод `A` (Value или Pointer) доступен через `B` (и как по значению, так и по указателю), так как `B` всегда содержит ссылку на `A`.

**Риск Pointer Embedding:** Если встроенная структура-указатель не инициализирована, вызов ее метода через внешнюю структуру приведет к `runtime panic`.

---

## Встраивание интерфейсов
Интерфейсы можно встраивать друг в друга для композиции контрактов (например, `io.ReadWriter`).
**Риск:** Если встроить интерфейс в структуру и не инициализировать его конкретной реализацией, вызов любого метода этого интерфейса приведет к панике.

## Вопросы для интервью (Senior/Staff)

### Как работает Method Set при встраивании?
Это один из самых частых вопросов. Правило S (Static) говорит: метод доступен, если тип его ресивера "совместим" с типом встраивания. Встраивание по указателю `*A` в `B` делает `B` удовлетворяющим всем интерфейсам, которые реализует `A`. Встраивание по значению — только тем, где ресиверы не указательные.

### Почему Composition лучше Inheritance в Go?
Композиция создает плоскую структуру и слабую связанность. Вы можете легко изменить реализацию встроенного типа без риска сломать логику всех "наследников". Это делает код более предсказуемым и пригодным для модульного тестирования.
