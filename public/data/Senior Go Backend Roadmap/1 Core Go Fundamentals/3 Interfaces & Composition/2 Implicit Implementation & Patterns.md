# 2. Implicit Implementation & Patterns

## Концепция: Implicit Implementation
Go использует структурную типизацию. Тип удовлетворяет интерфейсу, если он реализует все его методы. Отсутствие явного `implements` позволяет описывать контракты на стороне потребителя (consumer-defined interfaces).

### Преимущества для тестирования
Имплицитность позволяет изолировать любой сторонний пакет. Вы описываете только те методы, которые вам нужны, и создаете локальный интерфейс. Это позволяет легко подменять реальные зависимости моками без изменений в коде провайдера.

## Проектирование интерфейсов: Anti-patterns

### 1. Interface Pollution (Загрязнение интерфейсами)
Злоупотребление абстракциями ведет к потере читаемости кода и усложнению навигации (невозможность перейти к реализации в IDE одним кликом).
**Когда НЕ стоит использовать интерфейс:**
- Если существует только одна реализация.
- Если это преждевременная абстракция "на будущее".
- Внутри внутренних (internal) пакетов, где нет необходимости в публичном контракте.

### 2. Масштабируемость (Small vs Large)
Большие интерфейсы увеличивают связанность (coupling) и затрудняют их реализацию. Идеальный интерфейс в Go содержит 1-3 метода. Это позволяет легко комбинировать их (Composition) и соблюдать принцип подстановки Барбары Лисков.

---

## Pattern: Accept interfaces, return structs
Это фундаментальный принцип дизайна:
- Принимая интерфейс, вы делаете функцию максимально гибкой.
- Возвращая структуру, вы отдаете пользователю полный контроль над данными и методами типа, не ограничивая его абстракцией.

## Вопросы для интервью (Senior/Staff)

### Влияет ли интерфейс на производительность (Escape Analysis)?
Да. Работа через интерфейс почти всегда заставляет компилятор переносить данные в кучу (heap), так как он не может на этапе компиляции гарантировать время жизни объекта. Это увеличивает нагрузку на Garbage Collector (GC).

### Как избежать Interface Pollution при рефакторинге?
Начинать всегда со структур. Интерфейс должен появляться только тогда, когда действительно необходимо два и более варианта поведения (например, реальный клиент и тестовый мок) или когда необходимо скрыть реализацию в публичном API.
