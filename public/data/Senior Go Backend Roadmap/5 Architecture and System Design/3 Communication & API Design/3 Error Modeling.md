# 3. Error Modeling: Consistency across Services

## Проблема обработки ошибок
Если каждый микросервис возвращает ошибки в своем формате (один — строку, другой — JSON с `code`, третий — HTTP 500 на всё), фронтенд и другие сервисы превращаются в "спагетти" из условий.

## Стандартизация (Error Contract)
Нужен единый контракт для всей компании. Обычно это JSON структура:
```json
{
  "code": "insufficient_funds",
  "message": "Not enough money on balance",
  "request_id": "abc-123",
  "details": { "current_balance": 100, "required": 200 }
}
```

## Вопросы для интервью (Senior/Staff)

### Как прокидывать ошибки через gRPC, чтобы сохранить детали (Status Codes)?
В gRPC есть встроенный механизм `google.rpc.Status`. Он позволяет прикреплять к ошибке произвольные Protobuf-сообщения (Details), которые клиент может десериализовать и обработать программно (например, `RetryInfo` или `BadRequest`).

### Стоит ли возвращать Stack Trace клиенту в HTTP ответе?
**Категорически нет**. 
1. **Безопасность**: Стек-трейс раскрывает структуру проекта, пути к файлам и названия библиотек, что помогает злоумышленникам.
2. **UX**: Клиенту (фронту) не нужен ваш стек. Ему нужен понятный код ошибки и сообщение. Стек-трейс должен уходить только в логи и системы трейсинга (Sentry).

### Разница между Business Error и System Error
- **Business Error**: Ожидаемая ситуация (недостаточно денег, товар кончился). Это 4xx коды. Они должны быть типизированы.
- **System Error**: Неожиданный сбой (упала БД, таймаут сети). Это 5xx коды. Требуют алертов и вмешательства инженеров.

### Как реализовать глобальный обработчик (Middleware) для ошибок в Go?
Использовать паттерн `AppHandler`: хендлер возвращает не `void`, а `error`. Общий middleware перехватывает эту ошибку, проверяет её тип (через `errors.As`) и формирует стандартный JSON ответ с правильным HTTP статусом.
