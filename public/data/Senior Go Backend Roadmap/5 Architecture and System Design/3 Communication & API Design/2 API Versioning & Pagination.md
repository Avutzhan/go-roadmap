# 2. API Versioning & Pagination

## Стратегии версионирования
1. **URI Versioning**: `/v1/users`, `/v2/users`. (Самый популярный и явный метод).
2. **Header Versioning**: `Accept: application/vnd.myapi.v2+json`. (Чистый URI, но сложнее кешировать).
3. **Query Parameter**: `/users?version=2`. (Используется редко).

## Пагинация (Pagination)
**Staff требование**: Никогда не отдавайте список сущностей без лимитов.

### Методы:
1. **Offset Pagination**: `LIMIT 20 OFFSET 100`.
    - *Минус*: На больших смещениях (OFFSET 1.000.000) БД должна просканировать миллион строк, чтобы отдать 20. Это медленно (**O(N)**).
2. **Cursor (Keyset) Pagination**: `WHERE id > last_seen_id LIMIT 20`.
    - *Плюс*: Работает за константное время (**O(log N)** через индекс). Устойчива к удалению/добавлению записей между запросами.
    - *Минус*: Нельзя прыгнуть на "страницу 500".

## Вопросы для интервью (Senior/Staff)

### Как реализовать версионирование, если в `/v2` изменился только формат одного поля?
Паттерн **API Gateway / Sidecar**: запрос приходит на `/v2`, шлюз преобразует его в формат `/v1` (если бизнес-логика не менялась) и передает сервису. Это позволяет поддерживать старые версии без дублирования кода в коренном сервисе.

### В чем проблема Offset пагинации при частом обновлении данных?
Если пользователь читает первую страницу, а в этот момент в начало таблицы добавилась новая запись, то при переходе на вторую страницу пользователь увидит последний элемент первой страницы снова (дубликат). Cursor-пагинация лишена этой проблемы, так как она привязана к конкретному якорю в данных.

### Что такое "Breaking Change" в API и как их избегать?
Удаление поля, переименование поля, изменение типа данных, добавление обязательного аргумента.
**Как избегать**: 
- Только добавлять новые поля (Add-only).
- Помечать старые как `deprecated`.
- Делать новые поля опциональными.
- Если изменения радикальные — выпускать новую мажорную версию.

### Подход "Evolutionary API"
Вместо версий (v1, v2) — использование Protobuf и только аддитивных изменений. GraphQL также позволяет клиенту запрашивать ровно те поля, которые ему нужны, снижая риски при изменении схемы.
