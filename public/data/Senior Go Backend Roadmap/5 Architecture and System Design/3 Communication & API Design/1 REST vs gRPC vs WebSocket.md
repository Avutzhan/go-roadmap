# 1. REST vs gRPC vs WebSocket

## REST (Representational State Transfer)
- **Протокол**: HTTP/1.1 (чаще всего).
- **Формат**: JSON (текстовый).
- **Плюсы**: Идеальная совместимость с браузерами, кешируемость на уровне HTTP прокси, простота отладки (curl).
- **Минусы**: Оверхед на заголовки, отсутствие строгой типизации (если не использовать OpenAPI/Swagger).

## gRPC (Google Remote Procedure Call)
- **Протокол**: HTTP/2.
- **Формат**: Protobuf (бинарный).
- **Плюсы**: Очень высокая скорость, двунаправленные стримы, строгая типизация (IDL), генерация кода для 10+ языков.
- **Минусы**: Сложно тестировать из браузера, бинарный формат не читаем человеком без спецсредств.

## WebSocket
- **Протокол**: Upgrade из HTTP/1.1.
- **Цель**: Полнодуплексная связь в реальном времени.
- **Пример**: Чаты, биржевые котировки, уведомления.

## Вопросы для интервью (Senior/Staff)

### По каким критериям вы будете выбирать между REST и gRPC для внутреннего взаимодействия микросервисов?
Для внутренних вызовов Senior выберет **gRPC**: компактный бинарный формат экономит трафик и CPU на сериализации, а HTTP/2 позволяет мультиплексировать запросы в одном соединении. REST лучше оставить для публичного API (External API), так как он проще для сторонних разработчиков и интеграций.

### В чем преимущество HTTP/2 (над которым работает gRPC) перед HTTP/1.1?
1. **Мультиплексирование**: Множество запросов в одном TCP соединении без блокировки (Head-of-line blocking на уровне HTTP).
2. **Сжатие заголовков (HPACK)**: Заголовки в HTTP часто дублируются, HTTP/2 передает только дельту.
3. **Binary framing**: Данные передаются в бинарном виде, что легче парсить машинам.

### Почему Protobuf быстрее JSON?
JSON — это текст. Парсинг текста требует много операций с памятью и CPU. Protobuf — это бинарный формат с тегами. Он не хранит имена полей в сообщении (только их номера), и его структура заранее известна парсеру. Это позволяет выполнять десериализацию почти со скоростью копирования памяти.

### Когда WebSocket хуже, чем gRPC Streams или Server-Sent Events (SSE)?
SSE проще для реализации, если вам нужен только поток событий от сервера к клиенту (однонаправленный). WebSockets сложнее в масштабировании, так как они удерживают состояние на сервере (Stateful) и требуют "липких" сессий (Sticky Sessions) или внешнего Pub/Sub для синхронизации нод.
