# 5.9 Large-Scale Migrations (Staff Level)

Для уровня Staff / Principal Engineering навык безопасной миграции данных и архитектуры (с нуля или с легаси) — один из самых важных. От вас ожидают, что вы можете спроектировать этот процесс без даунтайма (zero-downtime) и без потери данных.

## 1. Zero-downtime migration & Dual Writes
Миграция базы данных или хранилища "на лету" обычно идет по паттерну **Dual Writes** (двойная запись).
Процесс из 4-5 шагов:
1. **Подготовка**: Развернуть новую БД.
2. **Шаг 1: Dual Write (Async)**. Приложение пишет в старую БД синхронно, а в новую — асинхронно или через Change Data Capture (Debezium/Kafka). Читает только из старой.
3. **Шаг 2: Backfill (Исторические данные)**. Перенос старых данных (миграционный скрипт, батчи). При этом надо следить за конфликтами: новые записи из шага 1 имеют приоритет над старыми.
4. **Шаг 3: Dual Write (Синхронно)**. Пишем в обе синхронно. Если в новой БД ошибка записи, решаем — фоллить запрос (soft) или это фатальная ошибка и откатываем транзакцию (strict).
5. **Шаг 4: Dark Read / Dark Traffic**. Читаем из обеих, логируем расхождения. Если данные совпадают на 99.999%, готовимся к переключению.
6. **Шаг 5: Switch Read**. Читаем из новой БД.
7. **Шаг 6: Stop Dual Write**. Прекращаем запись в старую БД и удаляем ее код.

**Интервью фокус**: Всегда упоминайте Debezium/CDC как альтернативу изменениям в коде монолита (dual write из кода уязвим к падениям ноды между двумя коммитами). 

## 2. Rollout Strategy & Compatibility
- **Backward/Forward Compatibility**: API и схемы БД всегда должны добавляться, но не модифицироваться.
  - *Expand and Contract (Parallel Change)*: Добавили новое поле -> читаем оба поля -> сделали новое основным -> удалили старое. Никогда нельзя делать `RENAME COLUMN` в работающем продакшне напрямую.
- **Rollout**: Canary Deployment. Сначала направляем 1% трафика на новую логику, следим за метриками и эррорами, затем 5%, 10%, 100%. Вы должны продумать метрики, по которым поймете, успешна ли миграция: Error Budget, P99 Latency.

## 3. Feature Flags
- Использование фича-флагов (LaunchDarkly, Unleash или кастом в Redis/AppConfig) для включения нового поведения без релиза.
- **Риски**: Утечки памяти на флагах, превращение кода в лапшу из `if`. Жизненный цикл флага: Создан -> Раскатан -> **УДАЛЕН из кода**. 

## 💬 Вопросы для интервью

1. **Как вы будете мигрировать терабайтную таблицу PostgreSQL на новую схему, не останавливая запись?**
   *Ответ: Expand and Contract паттерн. Добавлю новую колонку/таблицу, включу Dual Write через Debezium, потом Backfill скрипт мигрирует старые строки, замерю сходимость данных, переключу чтение.*

2. **Чем плоха двойная запись на уровне приложения? (Application-level dual write)**
   *Ответ: Нет распределенных транзакций. Если база 1 закоммитилась успешно, а приложение упало до коммита в базу 2, данные разъедутся. Решение — Outbox Pattern или Change Data Capture из WAL (Write-Ahead Log).*
