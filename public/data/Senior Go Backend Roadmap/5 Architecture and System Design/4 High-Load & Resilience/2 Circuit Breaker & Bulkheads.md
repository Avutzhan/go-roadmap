# 2. Circuit Breaker & Bulkheads

## Circuit Breaker (Предохранитель)
Паттерн для предотвращения каскадных сбоев. Если внешний сервис начал отвечать ошибками или таймаутами, мы "размыкаем цепь":
1. **Closed**: Всё хорошо, запросы идут.
2. **Open**: Порог ошибок превышен. Все запросы мгновенно отклоняются со своей локальной ошибкой, не доходя до внешнего сервиса. Даем сервису "отдохнуть".
3. **Half-Open**: Периодически пропускаем пробный запрос. Если он успешен — возвращаемся в Closed.

## Bulkheads (Переборки)
Заимствовано из судостроения. Разделение ресурсов (потоки, соединения, память) на независимые пулы для разных частей системы. Если упадет одна часть (например, поиск), остальная система (заказы) продолжит работать.

## Вопросы для интервью (Senior/Staff)

### Как настроить пороги для Circuit Breaker?
Это искусство компромисса. Обычно используют скользящее окно (Sliding Window) запросов (например, последние 100 запросов или 30 секунд). Если % ошибок > 50%, размыкаем. Staff-инженер должен учитывать "нормальный" фон ошибок системы.

### В чем опасность неправильно настроенного таймаута при наличии ретраев?
Это создает **Retry Storm**. Если сервис тормозит, клиент переподает запрос, увеличивая нагрузку на сервис еще сильнее, что делает его еще медленнее. Решение: таймаут должен быть жестким, а количество ретраев — минимальным с обязательным **Exponential Backoff**.

### Пример реализации Bulkhead в Go
Использование разных `http.Client` (с разными пулами соединений) для разных внешних сервисов. Если API оплаты зависнет и сожрет все 100 соединений своего пула, пул для API каталога останется свободным, и пользователи смогут хотя бы смотреть товары.

### Что такое "Graceful Degradation" (Мягкая деградация)?
Если основной сервис недоступен, мы предлагаем альтернативу.
*Пример*: Если сервис персональных рекомендаций упал, мы показываем пользователю список "Топ-продаваемых товаров" (кешированный статичный список). Это лучше, чем пустая страница или ошибка 500.
