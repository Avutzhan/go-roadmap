# 2. Consistent Hashing & CDC

## Консистентное хеширование (Consistent Hashing)
Алгоритм распределения данных, который минимизирует количество перемещаемых ключей при изменении количества серверов (шардов).
- **Суть**: Ключи и сервера располагаются на воображаемом "кольце" (хеш-пространстве). Ключ принадлежит ближайшему серверу по часовой стрелке.
- **Результат**: При добавлении сервера переезжает только `1/N` часть данных, а не всё подряд.

## Change Data Capture (CDC)
Процесс извлечения изменений из логов БД (WAL/Binlog).
- **Инструменты**: Debezium, Maxwell, Canal.
- **Сценарии**: Синхронизация Postgres с ElasticSearch, инвалидация кэша в Redis, наполнение DWH (ClickHouse).

## Вопросы для интервью (Senior/Staff)

### Зачем нужны виртуальные узлы (Virtual Nodes) в консистентном хешировании?
Чтобы избежать дисбаланса. Без них один участок кольца может быть слишком большим. Виртуальные узлы позволяют "размазать" один физический сервер по разным частям кольца, делая распределение нагрузки идеально равномерным.

### Почему CDC лучше, чем опросы по расписанию (Polling)?
1. **Нагрузка**: Polling нагружает БД запросами `SELECT`. CDC читает файл лога, это почти бесплатно.
2. **Real-time**: CDC дает задержку в миллисекунды.
3. **Удаления**: Polling не видит удаленные записи (если нет Soft Delete). CDC видит всё, что попало в лог транзакций.

### Как CDC помогает реализовать паттерн Transactional Outbox?
Вместо того чтобы приложение само писало в таблицу `outbox` и кто-то её читал, CDC может стримить все вставки в основную бизнес-таблицу напрямую в Kafka. Это упрощает код и делает доставку событий гарантированной на уровне инфраструктуры.

### Какова надежность CDC при сетевых сбоях?
Инструменты типа Debezium хранят свой собственный `offset` (позицию в логе БД). При рестарте они продолжат с того места, где остановились. Это обеспечивает гарантию доставки "At-least-once".
