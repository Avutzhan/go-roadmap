# 3. Query Optimization & Explain

## Как БД выполняет запрос
Планировщик строит дерево выполнения (Query Plan), выбирая между разными способами доступа к данным:
- **Seq Scan**: Полное сканирование таблицы. (Плохо для больших таблиц).
- **Index Scan**: Использование индекса для поиска адреса строки, затем чтение из кучи.
- **Index Only Scan**: Чтение данных только из индекса.
- **Bitmap Index Scan**: Оптимизация для множественных совпадений.

## EXPLAIN ANALYZE
- **EXPLAIN**: Показывает только план.
- **ANALYZE**: Реально выполняет запрос и показывает время каждой стадии.
**Staff фокус**: Смотрите на `Buffers` (количество прочитанных страниц из памяти и диска).

## Вопросы для интервью (Senior/Staff)

### Что такое "Cardinality" и почему она важна для планировщика?
Кардинальность — это ожидаемое количество уникальных значений. Если планировщик считает, что условие вернет 90% строк таблицы, он выберет `Seq Scan`, так как индекс будет только лишним оверхедом. Если статистика БД устарела, планировщик может выбрать неверный план.

### Как лечить проблему "N+1 Query"?
1. **SQL Join**: Объединить запросы.
2. **In-clause**: Собрать все ID и сделать один `WHERE id IN (...)`.
3. **Eager Loading**: В ORM настройка жадной загрузки связанных сущностей.

### В чем разница между Nested Loop Join и Hash Join?
- **Nested Loop**: Берет строку из одной таблицы и ищет совпадение в другой (хорошо, если вторая таблица маленькая или есть индекс).
- **Hash Join**: Строит хеш-таблицу в памяти из одной таблицы и прогоняет вторую через нее. Требует много памяти, но эффективен для больших наборов данных без индексов.

### Почему использование `SELECT *` — это антипаттерн?
1. **I/O**: Лишние данные из диска/памяти.
2. **Сеть**: Передача ненужных байт в приложение.
3. **Индексы**: Отключает возможность `Index Only Scan`.
4. **Контракт**: При добавлении колонки в БД логика приложения может сломаться.
